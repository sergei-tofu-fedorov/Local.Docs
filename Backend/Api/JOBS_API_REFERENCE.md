# Jobs API Reference

Complete reference for all Jobs API endpoints with request/response examples.

**Target Audience**: Frontend Developers, Backend Developers, QA Engineers

## Base Path

`/api/v3/jobs`

## Endpoints Overview

| Method | Endpoint | Description |
|--------|----------|-------------|
| PUT | `/api/v3/jobs` | [Upsert job](#1-upsert-job) |

| GET | `/api/v3/jobs/count` | [Get job count](#3-get-job-count) |
| GET | `/api/v3/jobs/stats` | [Get job statistics](#4-get-job-statistics) (counts per filter) |

## Write Operations

### 1. Upsert Job

Create or update a job. This is the only write endpoint for jobs.

**Endpoint**: `PUT /api/v3/jobs`

**Request Body**:

```json
{
  "id": "7d2a5c8a-6bf7-4c87-9b7c-1e3d8f4f5a12",
  "version": 1,
  "number": "42",
  "title": "Emergency Visit",
  "manualStatus": "completed",
  "clientId": "client_123",
  "invoiceId": "invoice_789",
  "estimateId": "estimate_456",
  "items": [
    {
      "name": "Faucet installation",
      "details": "Includes parts and labor",
      "itemType": "service",
      "description": "Kitchen faucet replacement",
      "unitPrice": 120.0,
      "unitType": "hour",
      "quantity": 2,
      "discount": {
        "type": "percent",
        "value": 10.0
      },
      "isTaxApplied": true,
      "catalogId": "item-001"
    }
  ],
  "visits": [
    {
      "id": "2b1c0c1f-52ee-4b9c-8b1a-71b6b25a9f3c",
      "dateTime": "2025-10-28T14:30:00Z",
      "assignedWorkerId": "worker_123",
      "status": "inProgress",
      "attachments": [
        {
          "id": "a1b2c3d4-0000-0000-0000-000000000001",
          "content": { "id": "content_abc" },
          "order": 0,
          "tags": ["Before"]
        }
      ]
    }
  ],
  "clientSnapshot": {
    "name": "Emily Johnson",
    "phone": "(310) 216-1600",
    "email": "emily.johnson@gmail.com",
    "address": "10848 Wagner St, Culver City, CA 90230"
  }
}
```

**Fields**:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | No | Job ID (GUID format). When omitted on create, a new ID is auto-generated by the client or server. When provided and job exists, updates the existing job. |
| `version` | integer | No | Job version for optimistic concurrency. Required on update to prevent conflicting writes. |
| `number` | string | No | Custom job number. When omitted on create, the system generates the next sequential number per account. |
| `title` | string | No | Job title. When omitted on create, defaults to the client name. Falls back to `Job #{number}` if client has no name. |
| `manualStatus` | string | No | Manual status override. Valid values: `none`, `completed`. When `none` or omitted, job status is computed from visits and invoice. |
| `clientId` | string | Yes | Reference to an existing client entity. |
| `invoiceId` | string | No | When provided, links the job to the specified invoice (`Invoice.JobId` is set to this job's id) and updates the job summary invoice status. Idempotent if already linked. |
| `estimateId` | string | No | When provided, links the job to the specified estimate (`Estimate.JobId` is set to this job's id). Idempotent if already linked. |
| `items` | array | No | Collection of job items. See [Item Structure](#reference-item-structure). Fully replaces existing items; `null` or omitted means "leave as is" on update. |
| `visits` | array | No | Replaces the job's visits collection. Each item has `id` (optional for new visits), `dateTime` (ISO 8601), optional `assignedWorkerId`, `status`, and optional `attachments` array. When `attachments` is provided, the server diffs against current state; `null` or omitted means "don't touch". See [Photos API Reference](../../features/jobs/implementation/10_photos/PHOTOS_API_REFERENCE.md) for attachment details. |
| `clientSnapshot` | object | No | Frozen client info to store with the job. Contains `name`, `phone`, `email`, `address` (all nullable strings). Typically sent when the job is marked as completed to preserve client details at that point in time. |

**Request Header** (optional):

| Header | Description |
|--------|-------------|
| `X-Client-Event-Time` | ISO 8601 timestamp for the client-side event time. Used for the initial job timeline event. When omitted, the backend uses current UTC time. |

**Response**: `200 OK`

Returns a **JobDto** with the created/updated job:

```json
{
  "id": "7d2a5c8a-6bf7-4c87-9b7c-1e3d8f4f5a12",
  "accountId": "acc_xyz789",
  "version": 1,
  "createdAt": "2025-10-28T08:00:00Z",
  "number": "42",
  "title": "Emergency Visit",
  "status": "inProgress",
  "manualStatus": "completed",
  "clientId": "client_123",
  "invoiceId": "invoice_789",
  "estimateId": "estimate_456",
  "visits": [
    {
      "id": "2b1c0c1f-52ee-4b9c-8b1a-71b6b25a9f3c",
      "jobId": "7d2a5c8a-6bf7-4c87-9b7c-1e3d8f4f5a12",
      "dateTime": "2025-10-28T14:30:00Z",
      "assignedWorkerId": "worker_123",
      "status": "inProgress",

      "attachments": [
        {
          "id": "a1b2c3d4-0000-0000-0000-000000000001",
          "type": "photo",
          "content": { "id": "content_abc", "url": "https://storage.googleapis.com/..." },
          "order": 0,
          "tags": ["Before"],
          "capturedAt": "2025-10-28T14:25:00Z",
          "createdAt": "2025-10-28T14:30:00Z"
        }
      ]
    }
  ],
  "items": [
    {
      "name": "Faucet installation",
      "details": "Includes parts and labor",
      "itemType": "service",
      "description": "Kitchen faucet replacement",
      "unitPrice": 120.0,
      "unitType": "hour",
      "quantity": 2,
      "discount": {
        "type": "percent",
        "value": 10.0
      },
      "isTaxApplied": true,

    }
  ],
  "currencyCode": "USD",
  "completionTime": null,
  "actionsRequired": [],
  "clientSnapshot": null

```

**Business Rules**:
- `clientId` is required for create and update.
- Creating a job for an **archived client** returns `400` with error code `clientArchived`. Updating an existing job for an archived client is allowed.
- The `visits` array fully replaces existing visits; an empty array removes all visits.
  - The `items` array fully replaces job items. Sending an empty or omitted `items` array clears all existing items (current behavior treats omitted and empty the same).
- Job status is computed from visit states and invoice status. Use `manualStatus` to manually override the computed status (e.g., mark as `completed`).
- If the job is soft-deleted, upsert returns a not-found error.
 - When `invoiceId` is provided and the invoice exists, its `jobId` is set to this job and the job summary is recalculated.
 - When `estimateId` is provided and the estimate exists, its `jobId` is set to this job.

## Read Operations

### 2. Get Paginated Jobs

Get jobs with pagination and filtering by display status.

**Endpoint**: `GET /api/v3/jobs/paged`

**Query Parameters**:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `limit` | integer | No | Items per page (1-100), default: 20 |
| `token` | string | No | Pagination cursor from previous response |
| `status` | string | No | Filter by display status (combines job status + invoice payment status). See values below. |
| `clientId` | string | No | Filter by client ID |

**Example Request**:

```
GET /api/v3/jobs/paged?limit=20&status=allOpen&clientId=client_123
```

**Display Status Filter Values**:

| Value | Description | Filter Logic |
|-------|-------------|--------------|
| `all` | All jobs (including paid) | No filter |
| `allOpen` | All jobs except paid | Job.Status != Paid |
| `scheduled` | Scheduled jobs awaiting work | Job.Status = Scheduled |
| `inProgress` | Jobs with work currently in progress | Job.Status = InProgress |
| `paymentDue` | Work completed, awaiting payment | Job.Status = ReadyForInvoice OR Job.Status = Invoiced |
| `unscheduled` | Jobs not yet scheduled | Job.Status = Unscheduled |
| `paid` | Paid jobs | Job.Status = Paid |

**Response**: `200 OK`

```json
{
  "items": [
    {
      "id": "7d2a5c8a-6bf7-4c87-9b7c-1e3d8f4f5a12",
      "number": "1",
      "title": "Kitchen Renovation",
      "status": "scheduled",
      "manualStatus": "none",
      "invoiceStatus": "unpaid",
      "createdAt": "2025-10-28T08:00:00Z",
      "completionTime": null,
      "invoiceId": "invoice789",
      "currencyCode": "USD",
      "totalDue": 1200.00,
      "totalAmount": 1200.00,
      "totalScope": 1200.00,
      "displayAmount": 1200.00,
      "actionsRequired": ["overdue"],
      "nextVisitDate": "2025-11-01T14:30:00Z"
    }
  ],
  "nextToken": "eyJpZCI6ImpvYjQ1NiJ9",
  "totalCount": null
}
```

**Response Fields**:

| Field | Type | Description |
|-------|------|-------------|
| `items` | array | List of job summaries |
| `items[].id` | Guid | Job identifier |
| `items[].number` | string | Job number |
| `items[].title` | string | Job title |
| `items[].status` | string | Job status (see [Job Status Values](#reference-job-status-values)) |
| `items[].manualStatus` | string | Manual status override: `none` or `completed` |
| `items[].invoiceStatus` | string | Invoice payment status: `unpaid` or `paid` |
| `items[].createdAt` | string (ISO 8601) | When job was created |
| `items[].completionTime` | string (ISO 8601) | When job was completed (nullable) |
| `items[].invoiceId` | string | Linked invoice ID (nullable) |
| `items[].currencyCode` | string | Currency code (nullable). From linked invoice when available, otherwise `"USD"`. |
| `items[].totalDue` | decimal | Invoice total due (nullable, only set when invoice is linked) |
| `items[].totalAmount` | decimal | Invoice total amount (nullable, only set when invoice is linked) |
| `items[].totalScope` | decimal | Sum of job items (per-item `unitPrice * quantity`, minus any item-level discount) |
| `items[].displayAmount` | decimal | Computed amount to display. See [Amount Display Rules](#amount-display-rules) |
| `items[].actionsRequired` | array | Actions needed for this job (see [Action Required Values](#reference-action-required-values)) |
| `items[].nextVisitDate` | string (ISO 8601) | Date of next closest planned visit - Scheduled or InProgress (nullable) |
| `nextToken` | string | Pagination cursor for next page (null if no more pages) |
| `totalCount` | integer | Total count (currently always null) |

### 4. Get Job Statistics

Get job counts and total amounts per filter status for the filter dropdown.

**Endpoint**: `GET /api/v3/jobs/stats`

**Example Request**:

```
GET /api/v3/jobs/stats
```

**Response**: `200 OK`

```json

      "count": 10,

      "count": 3,
      "totalsByCurrency": [

      ]
    },
    {
      "status": "inProgress",
      "count": 2,
      "totalsByCurrency": [
        { "currencyCode": "USD", "totalAmount": 3000.00 }
      ]
    },
    {
      "status": "paymentDue",
      "count": 4,
      "totalsByCurrency": [
        { "currencyCode": "USD", "totalAmount": 6590.00 }
      ]
    },
    {
      "status": "unscheduled",

      ]

      "count": 4,
      "totalsByCurrency": [
        { "currencyCode": "USD", "totalAmount": 11100.60 }
      ]
    }
  ]
}
```

**Response Fields**:

| Field | Type | Description |
|-------|------|-------------|
| `items` | array | Statistics for each filter status |
| `items[].status` | string | Filter status key (same values as paged endpoint filter) |
| `items[].count` | integer | Number of jobs matching this filter |
| `items[].totalsByCurrency` | array | Totals grouped by currency |
| `items[].totalsByCurrency[].currencyCode` | string | Currency code, e.g. `USD` |
| `items[].totalsByCurrency[].totalAmount` | decimal | Sum of display amounts in this currency (see [Amount Display Rules](#amount-display-rules)) |

**Business Rules**:

- Returns statistics for all 6 filter statuses in a single response
- Excludes soft-deleted jobs from all counts
- `totalAmount` per currency is computed per job using `GetDisplayAmount()` (see [Amount Display Rules](#amount-display-rules)), then summed
- Currency per job is determined by `GetDisplayCurrencyCode()` — uses `Job.DefaultCurrencyCode` (`"USD"`) when displaying Scope Total, `InvoiceAmounts.CurrencyCode` when displaying invoice amounts
- Jobs with different display currencies are aggregated separately within each status bucket
- Uses the same filter logic as the paged endpoint

### 5. Sync Jobs

Synchronize jobs using sequence-based pagination. Returns jobs that have been created, updated, or deleted since the last sync. Ideal for keeping a local cache in sync with the server.

**Endpoint**: `GET /api/v3/jobs/sync`

**Query Parameters**:

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `cursor` | string | No | Pagination cursor from previous response. Omit for initial sync. |
| `limit` | integer | No | Maximum items per page (default: 100) |
| `includeRelations` | boolean | No | Include related data (clients) in the response (default: `false`) |

**Example Request**:

```
GET /api/v3/jobs/sync?limit=50
```

**Response**: `200 OK`

**Response Fields**:

| Field | Type | Description |
|-------|------|-------------|
| `items` | array | List of sync change items |
| `items[].itemId` | string | Job identifier |
| `items[].change` | string | Change type: `updated` or `deleted` |
| `items[].item` | object | Full JobDto (null for deleted items). See [Job Object](#job-object-job) |
| `nextCursor` | string | Cursor for fetching next page (null if no more pages) |
| `hasMore` | boolean | Whether more items exist beyond this page |
| `delayNextRequestInSeconds` | integer | Recommended delay before next sync request (for rate limiting) |
| `relations` | object | Related data (only present when `includeRelations=true`). See below. |

**Relations Object** (when `includeRelations=true`):

| Field | Type | Description |
|-------|------|-------------|
| `clients` | array | Client objects for jobs in this page that don't have a `clientSnapshot`. If a client is not found, it is simply absent from the array. See [Client Structure](#reference-client-structure). |

**Change Types**:

| Value | Description |
|-------|-------------|
| `updated` | Job was created or modified. Full job data is included in `item`. |
| `deleted` | Job was soft-deleted. `item` is null. |

**Business Rules**:

- Returns changes in sequence order (oldest first within each page)
- Initial sync (no cursor) returns all non-deleted jobs
- Subsequent syncs return only changes since the cursor position
- Deleted jobs appear once in results, then are excluded from future syncs
- The `delayNextRequestInSeconds` value should be respected to avoid rate limiting
- Use `hasMore` to determine if more pages exist

## Reference: Job Status Values

Job status is computed from visit states and invoice status, or can be manually overridden.

| Value | API String | Description |
|-------|------------|-------------|
| `Unscheduled` | `unscheduled` | Job has no scheduled visits |
| `Scheduled` | `scheduled` | Job has at least one upcoming scheduled visit |
| `InProgress` | `inProgress` | Job has at least one visit currently in progress |
| `ReadyForInvoice` | `readyForInvoice` | All visits completed, no invoice yet |
| `Invoiced` | `invoiced` | All visits completed, invoice created but unpaid |
| `Paid` | `paid` | All visits completed, invoice fully paid |

**Status Computation Logic** (evaluated in order):

1. If `manualStatus` is `completed` (takes priority):
   - No invoice → `ReadyForInvoice`
   - Invoice unpaid → `Invoiced`
   - Invoice paid → `Paid`
2. If no visits → `Unscheduled`
3. If any visit is `InProgress` → `InProgress`
4. If any visit is `Scheduled` → `Scheduled`
5. If all visits are `Completed` but no manual override → `InProgress` (awaiting user confirmation)

**Note**: `manualStatus` is checked first, so a job without visits can still be completed by setting `manualStatus` to `completed`. It will then have status `ReadyForInvoice`, `Invoiced`, or `Paid` depending on invoice state.

## Reference: Action Required Values

Jobs may require user attention in certain situations. The `actionsRequired` array contains alerts computed at read time.

| Value | API String | Description |
|-------|------------|-------------|
| `Overdue` | `overdue` | A scheduled visit datetime has passed without being started |
| `AwaitingCompletion` | `awaitingCompletion` | All visits are completed but job is not marked complete |

**Detection Logic**:

- **Overdue**: Triggered when the earliest **Scheduled** visit datetime is in the past. Only `Scheduled` visits are considered for overdue - `InProgress` visits are not overdue (work has started), and `Completed` visits are not overdue (work is done).
- **AwaitingCompletion**: Triggered when the job has visits, all are completed, and the job's `manualStatus` is not `completed`.

Both conditions can be true simultaneously. The array contains all applicable alerts, or is empty if no action is required.

**Resolution**:

| Alert | Resolution Actions |
|-------|-------------------|
| `overdue` | Start visit, complete visit, reschedule, or cancel |
| `awaitingCompletion` | Mark job as complete, or schedule more visits |

## Reference: Invoice Status Values

The Jobs module uses two different invoice status representations:

### Full Invoice Status (Job Details)

When fetching a single job via `GET /api/v3/jobs/{id}`, the `invoice` object contains the full `InvoiceDto` with its native `status` field. This uses the standard invoice status values:

| Value | Description |
|-------|-------------|
| `Draft` | Invoice is in draft state |
| `Unpaid` | Invoice sent but not paid |
| `Paid` | Invoice fully paid |
| `PaidByCard` | Invoice paid via card |
| `PartiallyPaid` | Invoice partially paid |
| `Refunded` | Invoice refunded |
| `PartialRefunded` | Invoice partially refunded |

### Simplified Invoice Status (Paged Jobs)

For the paged jobs list (`GET /api/v3/jobs/paged`), the `invoiceStatus` field uses a binary paid/unpaid model for display purposes:

| Value | Description |
|-------|-------------|
| `unpaid` | Invoice is not fully paid (includes `Draft`, `Unpaid`, `PartiallyPaid`, `Refunded`, `PartialRefunded`) |
| `paid` | Invoice paid in full (`Paid` or `PaidByCard`) |

**Mapping Logic** (via `JobInvoiceStatusMapper`):
```csharp
public static JobInvoiceStatus FromInvoiceStatus(InvoiceStatus status) =>
    status is InvoiceStatus.Paid or InvoiceStatus.PaidByCard
        ? JobInvoiceStatus.Paid
        : JobInvoiceStatus.Unpaid;
```

The simplified `invoiceStatus` field appears in:
- `JobSummaryItemResponseDto.invoiceStatus` - stored in `JobSummaryView` and returned in paged queries

Filtering by invoice status uses the read model projection (`JobSummaryView`), so these queries do not make synchronous calls to the invoice module.

## Amount Display Rules

Jobs display different amounts depending on their effective status. This section describes **which amount to show** in jobs lists and statistics.

### Amount Sources

Jobs have two sources for amounts:

| Source | Description | Calculation |
|--------|-------------|-------------|
| **Scope Total** | Sum of job line items | `SUM(round(unitPrice * quantity) - round(discountValue))` for all job items |
| **Invoice Amount** | From linked invoice | `invoice.totalAmount` or `invoice.totalDue` |

### Display Rules

The `displayAmount` field is computed based on the job's **effective status**:

| Effective Status | Amount Shown |
|------------------|--------------|
| `invoiced` | `invoice.totalDue` |
| `paid` | `invoice.totalAmount` |
| All other statuses | Scope Total |

**Key principle**: Invoice amounts are only used for `invoiced` and `paid` statuses. All other statuses (including `readyForInvoice`) use **Scope Total**.

### Status-Based Logic

| Status | Description | Display Amount |
|--------|-------------|----------------|
| `unscheduled` | No visits scheduled | Scope Total |
| `scheduled` | Has scheduled visits | Scope Total |
| `inProgress` | Work in progress | Scope Total |
| `readyForInvoice` | Completed, no invoice | Scope Total |
| `invoiced` | Completed, invoice unpaid | Invoice Total Due |
| `paid` | Completed, invoice paid | Invoice Total Amount |

### Invoice Changes and Amount Switching

| Scenario | Result |
|----------|--------|
| Invoice deleted from completed job | Status changes to `readyForInvoice`, amount switches to Scope Total |
| Invoice created for completed job | Status changes to `invoiced`, amount switches to Invoice Total Due |
| Invoice paid | Status changes to `paid`, amount switches to Invoice Total Amount |
| Job completion undone | Status reverts, amount switches to Scope Total |

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| Invoice exists, job not completed | Scope Total (status is not `invoiced` or `paid`) |
| Invoice with $0 total | $0 displayed for `invoiced`/`paid` statuses |
| Job has no items | Scope Total = $0 |

### Implementation

The display amount is computed by `JobSummaryView.GetDisplayAmount()`:

```csharp
public decimal GetDisplayAmount()
{
    var totalScope = Job.CalculateTotalScope();

    if (Job.ManualStatus != JobManualStatus.Completed)
        return totalScope;

    if (string.IsNullOrEmpty(InvoiceId))
        return totalScope;

    return InvoiceStatus == JobInvoiceStatus.Paid
        ? InvoiceAmounts.TotalAmount ?? totalScope
        : InvoiceAmounts.TotalDue ?? InvoiceAmounts.TotalAmount ?? totalScope;
}
```

### Currency

Each job has a `CurrencyCode` property (default `"USD"`). When a job is created from an estimate, the estimate's currency is copied to the job.

The currency for each job's display amount depends on which amount source was used:

| Display Amount Source | Currency Source |
|----------------------|----------------|
| Scope Total | `Job.CurrencyCode` (from estimate, or `"USD"` default) |
| Invoice amount | `InvoiceAmounts.CurrencyCode` (from linked invoice) |

- Stats aggregate `displayAmount` grouped by the matching currency per job
- Default currency is `USD` when no currency is specified
- Jobs with different currencies are aggregated separately in stats
