Stage 5.7 - Invitation Per Worker
==================================

Three changes to the invitation system:
1. New endpoint to get a single invitation by ID (Auth)
2. BFF pass-through endpoint + error propagation (Invoices.Backend)
3. InvitationToken as DDD aggregate root — encapsulate magic token operations, remove separate repository

All parts are **implemented**.

---

## Part 1 — Get Invitation by ID

### Why

No endpoint existed to fetch a single invitation. The frontend relied on
`POST v1/tenants/{tenantId}/invitations/list` and filtered client-side.
A dedicated GET endpoint simplifies detail views and deep-links.

### Endpoint

```
GET v1/tenants/{tenantId}/invitations/{invitationId}
```

Authorization: required (same as other tenant invitation endpoints).

Response: `TenantInvitationResponse` (existing DTO, no changes needed).

```json
{
  "id": "guid",
  "tenantId": "string",
  "email": "string",
  "roleLevel": 2,
  "status": 0,
  "expiresAt": "2026-03-16T00:00:00Z",
  "createdAt": "2026-03-09T00:00:00Z",
  "acceptedAt": null,
  "revokedAt": null
}
```

Returns 404 (via `EntityNotFoundException`) if invitation not found or does
not belong to the tenant. `RevokeAsync` uses the same pattern.

> **Why 404 and not 400?** ID-based CRUD lookups (`GET /resource/{id}`,
> `POST .../revoke`) treat a missing resource as 404 per RFC 9110.
> Token-based business operations (accept, magic-login) keep 400 via
> `InvalidInvitationTokenException` because the endpoint exists but the
> input token is invalid.

### What Changed

| Layer | Change |
|-------|--------|
| Controller | `GetByIdAsync(tenantId, invitationId)` in `InvitationsController` |
| Service | `GetByIdAsync(tenantId, invitationId)` in `ITenantInvitationService` / `TenantInvitationService` |
| Repository | Reuses existing `ById(Guid id)` specification from `InvitationTokenSpecifications` |
| DTOs | None — reuses `TenantInvitationResponse` |
| Client | `GetTenantInvitationAsync(tenantId, invitationId)` in `ITofuAuthApiClient` / `TofuAuthApiClient` |
| Exceptions | `EntityNotFoundException` → 404 (replaces `InvalidInvitationTokenException` for ID-based lookups) |
| Tests | Functional tests in `Invitations.cs` (success + unauthorized + not-found-404), unit tests in `InvitationTokenTests.cs` |

## Part 1b — BFF Pass-Through (Invoices.Backend)

**Implemented.**

### Why

The Auth service now exposes `GET v1/tenants/{tenantId}/invitations/{invitationId}`.
The BFF needs a matching endpoint so the frontend can fetch a single invitation
without filtering the list client-side.

### Endpoint

```
GET api/invitations/{invitationId:guid}   [v3.0]
```

Authorization: required (same as other invitation endpoints).
Headers: `AccountId` required — used as `tenantId` when calling Auth service.

Response: `InvitationDto` (existing DTO, no changes needed).

```json
{
  "id": "guid",
  "accountId": "string",
  "email": "string",
  "role": "Worker",
  "status": "Pending",
  "createdAt": "2026-03-09T00:00:00Z",
  "expiresAt": "2026-03-16T00:00:00Z",
  "acceptedAt": null
}
```

Returns 404 when the invitation is not found, with error code
`entity_not_found` and the original Auth message forwarded to the client.

### Error Propagation: `entity_not_found`

Auth service returns `entity_not_found` (ProblemDetails, HTTP 404) for
ID-based lookups. The BFF needed a mechanism to propagate the original
status code and message to the client, instead of defaulting to 400 +
generic "Something went wrong on the auth server." message.

**Solution**: two-level exception conversion in `AuthApiExceptionConverter`:

1. **`KnownErrorTypes`** — a `HashSet<string>` of Auth error types that
   are converted to `InvoicesAuthApiException`. Unknown types return
   `null` (re-thrown as raw `AuthApiException`). This is the same set
   of error types that was previously in the `if` chain.

2. **`PropagateToClientErrorTypes`** — a subset of `KnownErrorTypes`
   whose original HTTP status code and message are forwarded to the
   client. Currently contains only `AuthErrorTypes.EntityNotFound`.

`InvoicesAuthApiException` gained a `PropagateToClient` property
(defaults to `false`, backward-compatible). The middleware checks this
flag to decide whether to use `exception.Title` or the generic message:

```csharp
result.Message = exception.PropagateToClient
    ? exception.Title
    : "Something went wrong on the auth server.";
```

For `entity_not_found`: client receives `404` + `"InvitationToken 'abc' was not found"`.
For all other known errors: client receives `400` + generic message (unchanged).

### What Changed

| Layer | Change |
|-------|--------|
| NuGet | `Tofu.Auth.Api.Client` updated to `0.6.0-preview001` (3 projects) |
| Decorator | `GetTenantInvitationAsync` pass-through in `TofuAuthApiClientDecorator` |
| Exception model | `InvoicesAuthApiException.PropagateToClient` property (default `false`) |
| Exception converter | Refactored to `KnownErrorTypes` + `PropagateToClientErrorTypes` HashSets; `EntityNotFound` propagates status code + message |
| Middleware | `BuildResultFromInvoicesAuthException` uses `exception.Title` when `PropagateToClient` is true |
| Controller | `GetByIdAsync(invitationId)` in `InvitationsController` |
| DTOs | None — reuses `InvitationDto` and `ToInvitationResponseDto()` |
| Tests | `GetById_ShouldReturnInvitation` + `GetById_NotFound_ShouldReturn404` (asserts 404 status, error code, message) |
| Test helpers | `MockSetup.CreateEntityNotFoundException(entityType, entityId)` — reusable factory |

### Flow

**Success path:**
```
InvitationsController.GetByIdAsync(invitationId)
  → ITofuAuthApiClient.GetTenantInvitationAsync(AccountId, invitationId)
  → TenantInvitationResponse.ToInvitationResponseDto()
  → InvitationDto
```

**Error path (invitation not found):**
```
ITofuAuthApiClient throws AuthApiDetailedException (entity_not_found, 404)
  → TofuAuthApiClientDecorator.ExecuteWithExceptionConversionAsync
  → AuthApiExceptionConverter: KnownErrorTypes ✓, PropagateToClientErrorTypes ✓
  → InvoicesAuthApiException (status=404, propagateToClient=true)
  → ApiExceptionHandlingMiddleware.AuthApiExceptionMap returns 404 + original title
```

Thin pass-through: no business logic in the BFF layer, just client call +
DTO mapping with existing code.

## Part 2 — InvitationToken as Aggregate Root

### Why

`InvitationMagicToken` is always accessed in the context of its parent
`InvitationToken`, yet it had its own repository
(`IInvitationMagicTokenRepository`). Services had to coordinate both
repositories in transactions. Making `InvitationToken` the aggregate
root lets all magic token operations go through the invitation —
removing the need for a separate repository.

### DDD Patterns Applied

- **Encapsulated collection**: private `List<InvitationMagicToken>` backing
  field with `IReadOnlyCollection` public accessor — external code cannot
  modify the collection directly.
- **State guards**: all mutating methods (`MarkAsAccepted`, `Revoke`,
  `AddMagicToken`, `UseMagicToken`) call `EnsurePending()` which throws
  `InvitationNotPendingException` if the invitation is not in Pending status.
- **Internal child operations**: `InvitationMagicToken.Create()` and
  `MarkAsUsed()` are `internal` — only the aggregate root (and test
  projects via `InternalsVisibleTo`) can call them.
- **Explicit `UpdatedAt`**: set in each mutating method of the aggregate root.

### InvitationToken — Aggregate Root

```csharp
private readonly List<InvitationMagicToken> _magicTokens = [];
public IReadOnlyCollection<InvitationMagicToken> MagicTokens => _magicTokens.AsReadOnly();

public string AddMagicToken(TimeSpan ttl)
{
    EnsurePending();
    var (magicToken, rawToken) = InvitationMagicToken.Create(this, ttl);
    _magicTokens.Add(magicToken);
    UpdatedAt = DateTimeOffset.UtcNow;
    return rawToken;
}

public void UseMagicToken(string rawToken)
{
    // Validates: not empty, pending status, token exists, not used, not expired
    // Throws MagicLoginException with appropriate reason on failure
    // Calls magicTokenEntity.MarkAsUsed() on success
}
```

### EF Configuration

Relationship defined on the `InvitationToken` side with backing field access:

```csharp
builder.HasMany(i => i.MagicTokens)
    .WithOne(m => m.Invitation)
    .HasForeignKey(m => m.InvitationId)
    .OnDelete(DeleteBehavior.Cascade);

builder.Navigation(i => i.MagicTokens)
    .UsePropertyAccessMode(PropertyAccessMode.Field);
```

The `InvitationMagicToken` configuration no longer defines the
relationship (owned by the parent side).

### Repository Changes

Both `FindOne` and `Find` eagerly load magic tokens via a shared
`QueryWithAllIncludes` property that includes `.Include(i => i.Role)` and
`.Include(i => i.MagicTokens)`. This ensures magic tokens are always
available when domain methods operate on the aggregate.

No `FindByMagicTokenHashAsync` was needed — the service finds the
invitation by its own token hash, and `UseMagicToken` validates the
magic token within the loaded collection.

**Deleted:**
- `IInvitationMagicTokenRepository` (interface)
- `InvitationMagicTokenRepository` (implementation)
- Registration removed from `DependencyInjection.cs`

### Service Changes

**TenantInvitationService.CreateAsync** — removed `_magicTokenRepository`,
uses aggregate method:

```
1. Validate tenant, role, rate limit (unchanged)
2. Create InvitationToken (unchanged)
3. invitation.AddMagicToken(magicTtl)  → returns raw token string
4. Save invitation (EF cascades the magic token insert)
5. Build link, send email, return
```

**InvitationProcessingService.ExchangeMagicTokenAsync** — removed
`_magicTokenRepository`, uses aggregate method:

```
1. Find invitation by token hash (loads magic tokens via Include)
2. invitation.UseMagicToken(rawMagicToken)  → validates + marks used
3. Save invitation (EF cascades the magic token update)
```

### Test Support

`InternalsVisibleTo` in `Tofu.Auth.Domain.csproj` is limited to
`Tofu.Auth.Domain.UnitTests` only. Functional tests seed magic tokens
through the public `invitation.AddMagicToken(ttl)` method and verify
results by loading the invitation with its `MagicTokens` collection —
no access to `internal` members is needed.

### What Changed

| Area | Change |
|------|--------|
| `InvitationToken` model | Backing field `_magicTokens`, `IReadOnlyCollection` accessor, `AddMagicToken()`, `UseMagicToken()`, `EnsurePending()` guard |
| `InvitationMagicToken` model | `Create()` and `MarkAsUsed()` changed to `internal` |
| `IInvitationMagicTokenRepository` | Deleted |
| `InvitationMagicTokenRepository` | Deleted |
| EF `InvitationTokenConfiguration` | `HasMany`/`WithOne` + `PropertyAccessMode.Field` |
| EF `InvitationMagicTokenConfiguration` | Removed relationship (now parent-owned) |
| `TenantInvitationService` | Removed `_magicTokenRepository` dependency |
| `InvitationProcessingService` | Removed `_magicTokenRepository`, uses `invitation.UseMagicToken()` |
| DI registration | Removed `IInvitationMagicTokenRepository` binding |
| Domain csproj | `InternalsVisibleTo` for `Tofu.Auth.Domain.UnitTests` only |

### What This Preserves

| Concern | Status |
|---------|--------|
| `InvitationMagicToken` entity and table | Unchanged — no migration needed |
| Magic token hashing and validation | Same logic, now inside aggregate |
| Invitation creation flow | Same behavior, fewer repos involved |
| Magic link exchange | Same validation, single aggregate lookup |
| Old magic tokens | Still expire naturally (12h TTL) |
