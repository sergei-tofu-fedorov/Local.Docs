# Step 5.3: Worker Deletion — Implementation Plan

> Clean up visit assignments when a worker is removed from the team.

## Motivation

When a team member is removed via `DELETE /api/team/members/{userId}`,
the system revokes their Tofu.Auth access and removes their local
`OwnedAccount` record — but **visits assigned to that worker are left
unchanged**. This creates orphaned `AssignedWorkerId` references:

- Worker name resolution returns `null` for the removed ID
- The visit appears assigned to a ghost user in the UI
- No activity timeline entry indicates the unassignment
- If the worker is re-added later, old visits silently reappear

### Current removal flow

**File:** `Invoices.Api/Controllers/TeamController.cs`

```
DELETE /api/team/members/{userId}
  1. _tofuAuthApiClient.RemoveUserFromTenantAsync(accountId, userId)  → revoke Auth access
  2. _masterUserRepository.RemoveMemberFromAccount(userId, accountId)  → remove OwnedAccount
  3. return 204 NoContent
```

No visit cleanup. No domain events. No cascade.

### Goals

1. **Unassign all visits** for the removed worker across all jobs in the account
2. **All visit statuses** — unassign regardless of status (Scheduled, InProgress, Completed)
3. **No events for now** — `VisitWorkerChanged` events may be added in a future iteration

## Phase 1: Command & Handler

### 1.1 Create `UnassignWorkerFromAllVisitsCommand`

**File:** `Jobs/Jobs.Contracts/Jobs/Commands/UnassignWorkerFromAllVisitsCommand.cs` (new)

```csharp
public sealed record UnassignWorkerFromAllVisitsCommand(
    string AccountId,
    string WorkerId,
    string? MasterUserId)
    : CommandBase(AccountId, MasterUserId), ICommand<UnassignWorkerFromAllVisitsResult>;

public sealed record UnassignWorkerFromAllVisitsResult;
```

### 1.2 Create `UnassignWorkerFromAllVisitsCommandHandler`

**File:** `Jobs/Jobs.Application/Commands/UnassignWorkerFromAllVisitsCommandHandler.cs` (new)

Steps:
1. Query all jobs in the account that have visits with
   `AssignedWorkerId == command.WorkerId`
2. For each affected job: call `Job.UnassignWorkerFromAllVisits(workerId)`
   which sets `AssignedWorkerId = null` on each matching visit
3. Save all modified jobs

No domain events are emitted — this is a cleanup operation, not a
user-initiated assignment change. No team validation is needed because
the worker is being removed; just null out the visit references.

**Repository query** (new method needed):

```csharp
Task<IReadOnlyList<Job>> GetJobsByAssignedWorker(
    string accountId, string workerId, CancellationToken ct);
```

### 1.3 Batch considerations

A worker could be assigned to many visits. The handler should:
- Load jobs in batches (not all at once) if the count is large
- Use a single `IUnitOfWork` transaction per batch
- Log the number of affected visits for observability

## Phase 2: Wire Into Team Removal Flow

### 2.1 Update `TeamController.RemoveTeamMember`

**File:** `Invoices.Api/Controllers/TeamController.cs`

Add the visit cleanup step before Auth revocation and local DB removal:

```csharp
[HttpDelete("members/{userId:guid}")]
public async Task<IActionResult> RemoveTeamMember(Guid userId, CancellationToken ct)
{
    var memberMasterUserId = userId.ToString().ToLowerInvariant();

    // 1. Unassign all visits
    var command = new UnassignWorkerFromAllVisitsCommand(AccountId, memberMasterUserId, AuthenticationInfo?.MasterUser?.Id);
    await _dispatcher.DispatchCommand<UnassignWorkerFromAllVisitsCommand, UnassignWorkerFromAllVisitsResult>(command, ct);

    // 2. Revoke Auth access
    await _tofuAuthApiClient.RemoveUserFromTenantAsync(AccountId, userId, ct);

    // 3. Remove local record
    await _masterUserRepository.RemoveMemberFromAccount(memberMasterUserId, AccountId, ct);

    return NoContent();
}
```

**Order matters:**
- Unassign visits **before** Auth revocation and local DB removal
- Auth revocation **before** local DB removal

### 2.2 Handle partial failure

If visit unassignment succeeds but Auth revocation fails:
- Visits are unassigned (safe — can be reassigned)
- Worker still has access (retry removal)

If Auth revocation succeeds but local DB removal fails:
- Worker has no Auth access (cannot log in)
- Local record is stale (retry or manual cleanup)

**Recommendation:** Wrap in a try/catch per step with logging.
Do not use a distributed transaction — eventual consistency is acceptable.

## Phase 3: Domain — `Job.UnassignWorkerFromAllVisits`

**File:** `Jobs/Jobs.Domain/Models/Job.cs`

Simple cleanup method — no domain events, no team validation, just
nulls out `AssignedWorkerId` on matching visits:

```csharp
/// <summary> Removes the worker from all visits assigned to them. Does not emit events. </summary>
public int UnassignWorkerFromAllVisits(string workerId)
{
    if (Visits == null) return 0;

    var count = 0;
    foreach (var visit in Visits.Where(v => v.AssignedWorkerId == workerId))
    {
        visit.SetAssignedWorker(null);
        count++;
    }

    return count;
}
```

Returns the number of visits unassigned (used for logging in the handler).

### Corner Cases

Worker is unassigned from **all** visits regardless of status — no blocking,
no guard, no special exception. Already-null visits are skipped.

| Case | Behavior |
|------|----------|
| **In-progress visit** | Unassign worker, keep `InProgress` status. Admin can reassign later. |
| **Completed visit** | Unassign worker. Historical data is preserved in the visit record. |
| **Already-null worker** | Skip — no update needed. |
| **No assigned visits** | Handler completes successfully — no-op. |
| **Deleted job** | Repository filters out deleted jobs; never returned by query. |
| **Concurrent assignment** | If someone assigns the worker to a new visit during removal, that visit won't be caught. Acceptable — eventual consistency. |
| **Multiple visits in same job** | All matched visits unassigned in a single job save. |
| **Stale mobile upsert** | Mobile sends `PUT /api/v3/jobs` with the deleted worker still assigned. `UpdateVisits` silently clears `assignedWorkerId` to `null` instead of throwing `404 workerNotFound`. This prevents stale client data from blocking the entire job upsert. |
