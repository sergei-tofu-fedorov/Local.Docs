# Step 5.x: Worker Permissions â€” Implementation Plan

> Enable role-based access control so Admin and Worker roles have different
> endpoint access and data visibility.

## Motivation

The codebase already stores user roles (`TenantRole` in `OwnedAccount`) and
retrieves permissions from the Auth API (`GetMyPermissionsAsync`), but
**nothing enforces these roles at the API level**. Every authenticated user
can call any endpoint. This step activates RBAC so workers are restricted
to their assigned work.

### Current state

| Aspect | State |
|--------|-------|
| Role storage | `OwnedAccount.TenantRole`: `null` = Admin, `"Worker"` = Worker |
| Role enum | `RoleLevelDto { Unknown, Admin, Worker }` |
| Auth API permissions | `GET /permissions` returns `Ability[]` â€” not enforced server-side |
| Authorization middleware | Code exists (`AuthorizationInstaller.cs`) but **disabled** in DI |
| Endpoint protection | All `/api/*` endpoints require authentication only â€” no role check |
| Worker data filtering | `WorkerController` filters visits by `assignedWorkerId` â€” functional but not enforced |

### Goals

1. **Enforce RBAC** â€” workers cannot call admin-only endpoints
2. **Data isolation** â€” workers see only their assigned visits/jobs
3. **Extensible** â€” new roles can be added without restructuring
4. **Auth API as source of truth** â€” roles and permissions come from Tofu.Auth

## Phase 1: Role Resolution in Middleware

### 1.1 Extract role from authenticated user context

**File:** `Invoices.Api/Middleware/AccountAuthenticationMiddleware.cs`

After successful Auth API authentication, the middleware already stores
`AuthenticationInfo` (which includes `MasterUser` with `OwnedAccounts`).

Add: resolve the user's role for the current account and store it in
`HttpContext.Items`:

```csharp
var account = authInfo.MasterUser?.OwnedAccounts
    ?.FirstOrDefault(a => a.AccountId == accountId);
var role = account?.TenantRole == "Worker"
    ? RoleLevel.Worker
    : RoleLevel.Admin;

context.Items[RoleLevelItemKey] = role;
```

### 1.2 Add `RoleLevel` to `BaseController`

**File:** `Invoices.Api/Controllers/BaseController.cs`

```csharp
protected RoleLevel RoleLevel =>
    HttpContext.Items.TryGetValue(RoleLevelItemKey, out var role)
        ? (RoleLevel)role
        : RoleLevel.Admin; // fallback for legacy auth
```

### 1.3 Define `RoleLevel` enum (internal)

**File:** `Invoices.Api/Authorization/RoleLevel.cs` (new or reuse `RoleLevelDto`)

```csharp
public enum RoleLevel
{
    Admin,
    Worker
}
```

## Phase 2: Authorization Policies

### 2.1 Define policies

**File:** `Invoices.Api/Authorization/AuthorizationInstaller.cs`

```csharp
public static class Policy
{
    public const string AdminOnly = nameof(AdminOnly);
    public const string WorkerOrAdmin = nameof(WorkerOrAdmin);
}
```

### 2.2 Create `RoleAuthorizationHandler`

**File:** `Invoices.Api/Authorization/RoleAuthorizationHandler.cs` (new)

A lightweight `IAuthorizationHandler` that reads `RoleLevel` from
`HttpContext.Items` and evaluates against the policy requirement.

```csharp
public class RoleRequirement : IAuthorizationRequirement
{
    public RoleLevel MinimumRole { get; init; }
}

public class RoleAuthorizationHandler : AuthorizationHandler<RoleRequirement>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        RoleRequirement requirement)
    {
        var httpContext = (context.Resource as HttpContext)
            ?? throw new InvalidOperationException();

        if (httpContext.Items.TryGetValue(RoleLevelItemKey, out var roleObj)
            && roleObj is RoleLevel role
            && role <= requirement.MinimumRole)
        {
            context.Succeed(requirement);
        }

        return Task.CompletedTask;
    }
}
```

### 2.3 Register policies in DI

**File:** `Invoices.Api/DI/ApiCommonConfiguration.cs`

Uncomment and update `ConfigureAuthorization()`:

```csharp
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy(Policy.AdminOnly, policy =>
        policy.AddRequirements(new RoleRequirement { MinimumRole = RoleLevel.Admin }));

    options.AddPolicy(Policy.WorkerOrAdmin, policy =>
        policy.AddRequirements(new RoleRequirement { MinimumRole = RoleLevel.Worker }));
});

builder.Services.AddSingleton<IAuthorizationHandler, RoleAuthorizationHandler>();
```

## Phase 3: Apply Policies to Controllers

### 3.1 Admin-only endpoints

Apply `[Authorize(Policy = Policy.AdminOnly)]` to endpoints that workers
must not access:

| Controller | Endpoint | Reason |
|------------|----------|--------|
| `JobsController` | `Upsert` (PUT) | Workers cannot create/edit jobs |
| `JobsController` | `Delete` | Workers cannot delete jobs |
| `ClientsController` | All mutating endpoints | Workers don't manage clients |
| `TeamController` | All endpoints | Workers don't manage team |
| `InvitationsController` | `CreateInvitation` | Workers cannot invite |
| `InvoicesController` | All endpoints | Workers don't manage invoices |

### 3.2 Worker-accessible endpoints

Apply `[Authorize(Policy = Policy.WorkerOrAdmin)]` or leave default auth:

| Controller | Endpoint | Notes |
|------------|----------|-------|
| `WorkerController` | All endpoints | Already filters by `assignedWorkerId` |
| `JobsController` | `GetPaged`, `GetDetails` | See Phase 4 for data filtering |
| `AuthorizationController` | `GetMyPermissions` | Both roles need this |

### 3.3 Decide: should workers access `JobsController` read endpoints?

**Open question:** Workers currently use `WorkerController` for their
visit-centric view. Should they also be able to call `JobsController`
read endpoints (with filtered results)?

**Option A:** Block workers from `JobsController` entirely â€” they use
`WorkerController` only.

**Option B:** Allow read-only access to `JobsController` with automatic
data filtering (only jobs with assigned visits).

**Recommendation:** Option A for initial implementation. Simpler to enforce,
`WorkerController` already provides the worker-specific view.

## Phase 4: Worker Data Filtering (WorkerController)

### 4.1 Verify existing filtering

**File:** `Invoices.Api/Controllers/WorkerController.cs`

The `WorkerController` already filters visits by `assignedWorkerId` matching
the authenticated user. Verify this uses `AuthenticationInfo.MasterUserId`
(not a client-provided value) so workers cannot request other workers' data.

### 4.2 Enforce `workerId` from auth context

If any query currently accepts `workerId` as a parameter, replace with
the authenticated user's ID:

```csharp
var workerId = AuthenticationInfo?.MasterUserId
    ?? throw new UnauthorizedAccessException();
```

Workers must never be able to pass an arbitrary `workerId` to see
another worker's visits.

## Phase 5: Financial Data Visibility

### 5.1 Decide: should workers see financial details?

**Open question from overview.md:** Should workers see job financial
details (amounts, invoices)?

**Recommendation:** No. Workers focus on visit execution (status, schedule,
location). Financial data is admin-only.

### 5.2 Strip financial fields from worker responses

**File:** Worker DTOs and mapping

If financial fields are currently included in `WorkerVisitDetailsDto`,

- `job.Items[].Rate`, `job.Items[].Amount` (keep description/quantity for work context)
- `job.InvoiceId`, `job.PaymentStatus`

If worker DTOs already exclude these fields â€” verify and document.

## Phase 6: Error Handling

### 6.1 Return 403 for unauthorized role access

When a worker hits an admin-only endpoint, ASP.NET Core authorization
returns `403 Forbidden` by default. Verify the error response format
matches the API's standard error envelope:

```json
{
    "errorCode": "forbidden",
    "message": "Insufficient permissions for this action"
}
```

### 6.2 Add `Forbidden` error code

**File:** `Invoices.Api/Middleware/ErrorCode.cs`

```csharp
[EnumMember(Value = "forbidden")]
Forbidden,
```

### 6.3 Custom 403 response handler

**File:** `Invoices.Api/DI/ApiCommonConfiguration.cs`

Configure authorization to return the standard error envelope instead
of a bare 403:

```csharp
builder.Services.AddAuthorization(/* ... */);

// In middleware pipeline:
app.UseStatusCodePages(context =>
{
    if (context.HttpContext.Response.StatusCode == 403)
    {
        // Write standard error envelope
    }
    return Task.CompletedTask;
});
```

## Phase 7: Permissions Sync with Auth API

### 7.1 Keep `GetMyPermissions` endpoint

**File:** `Invoices.Api/Controllers/AuthorizationController.cs`

The existing `GET /permissions` endpoint returns abilities from Auth API.
Clients (web/mobile) use this to show/hide UI elements. Keep as-is.

### 7.2 Server-side enforcement vs client-side hints

| Layer | Mechanism | Purpose |
|-------|-----------|---------|
| Server | `[Authorize(Policy)]` attributes | Hard enforcement â€” cannot bypass |
| Client | `GET /permissions` â†’ ability list | UI hints â€” show/hide buttons |

Both layers are needed. Server enforcement is the security boundary;
client permissions improve UX by preventing users from seeing actions
they can't perform.
