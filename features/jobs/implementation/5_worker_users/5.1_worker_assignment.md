# Step 5.1: Worker Assignment — Implementation Plan

> Worker assignment validation, `visitWorkerChanged` event emission,
> and team membership checks in the existing job upsert flow.

## Motivation

Assigning a worker to a visit happens through the standard job upsert
(`PUT /api/v3/jobs`) which replaces the entire visits array. The upsert
handles assignment mechanically but does not:

- Validate that the worker belongs to the account team
- Emit a dedicated assignment activity event (assignment is buried in `VisitCreated`)

### Current state

| Aspect | State |
|--------|-------|
| `Visit.AssignedWorkerId` | `string?` on domain model — stores worker ID |
| Assignment flow | `PUT /api/v3/jobs` (full upsert) — sets `assignedWorkerId` per visit |
| Worker validation | Name resolution only (`IJobWorkerService.GetWorkerName`) — no team membership check |
| Activity events | Assignment changes included in `VisitCreated` — no dedicated event |
| Unassignment | Set `assignedWorkerId` to `null` in upsert payload |

### Goals

1. **Worker validation** — verify worker exists and belongs to account team before accepting assignment
2. **`visitWorkerChanged` event** — emit `VisitWorkerChanged` domain event for assign, reassign, and unassign, matching [Job Timeline Events.csv](../../Job%20Timeline%20Events.csv)
3. **Activity timeline** — assignment changes appear as dedicated entries in the job timeline

## Phase 1: Domain Event

### 1.1 Add `VisitWorkerChanged` domain event

Matches the `visitWorkerChanged` event type defined in
[Job Timeline Events.csv](../../Job%20Timeline%20Events.csv).

**File:** `Jobs/Jobs.Domain/Events/VisitWorkerChanged.cs` (new)

```csharp
/// <summary> Raised when the assigned worker on a visit changes. </summary>
public sealed record VisitWorkerChanged(
    Guid JobId,
    Guid VisitId,
    VisitWorkerChangedPayload.WorkerInfo? PreviousWorker,
    VisitWorkerChangedPayload.WorkerInfo? NewWorker,
    DateTimeOffset OccurredAt) : IDomainEvent;
```

**Payload** (mirrors CSV structure):

```csharp
public static class VisitWorkerChangedPayload
{
    public sealed record WorkerInfo(string WorkerId, string? WorkerName);
}
```

| Scenario | `previousWorker` | `newWorker` |
|----------|-------------------|-------------|
| Assign (no previous) | `null` | `{ workerId, workerName }` |
| Reassign | `{ prevId, prevName }` | `{ newId, newName }` |
| Unassign | `{ prevId, prevName }` | `null` |

## Phase 2: Domain Model — Team & Assignment

### 2.1 Create `Team` value object

**File:** `Jobs/Jobs.Domain/Models/Team.cs` (new)

A lightweight value object representing the account's team, passed into
Job aggregate methods. Carries member roles alongside names so the
domain can enforce role-based rules (e.g., only workers can be assigned
to visits).

```csharp
/// <summary> Represents the account team for validation and name resolution. </summary>
public class Team
{
    private readonly IReadOnlyDictionary<string, TeamMember> _members;

    public Team(IEnumerable<TeamMember> members)
    {
        _members = members.ToDictionary(m => m.Id);
    }

    /// <summary> Returns the member or null if not found. </summary>
    public TeamMember? FindMember(string memberId) =>
        _members.GetValueOrDefault(memberId);

    /// <summary> Returns the member or throws WorkerNotFoundException. </summary>
    public TeamMember GetWorkerOrThrow(string workerId)
    {
        var member = FindMember(workerId)
            ?? throw new WorkerNotFoundException(workerId);

        if (member.Role != TeamMemberRole.Worker)
            throw new WorkerNotFoundException(workerId);

        return member;
    }
}

public sealed record TeamMember(string Id, string? Name, TeamMemberRole Role);

public enum TeamMemberRole
{
    Admin,
    Worker
}
```

**Usage in domain methods:**
- **Validation** — `team.GetWorkerOrThrow(workerId)` checks both
  existence and role (must be `Worker`)
- **Name resolution** — `team.FindMember(id)?.Name` for event payloads
  (previous worker may have been removed, so `FindMember` returns null
  gracefully)

### 2.2 Add `ChangeWorkerAssignment` method to `Job`

**File:** `Jobs/Jobs.Domain/Models/Job.cs`

A single method handles assign, reassign, and unassign. When `newWorkerId`
is non-null, the worker is validated via `team.GetWorkerOrThrow()`. When
`newWorkerId` is null, it unassigns (no role validation needed — the worker
is being removed).

```csharp
/// <summary> Changes the assigned worker on a visit: assign, reassign, or unassign. </summary>
public void ChangeWorkerAssignment(
    Guid visitId,
    string? newWorkerId,
    Team team,
    DateTimeOffset now)
{
    var visit = Visits.SingleOrDefault(v => v.Id == visitId)
        ?? throw new VisitNotFoundException(visitId);

    var previousWorkerId = visit.AssignedWorkerId;

    if (previousWorkerId == newWorkerId)
        return; // no change

    if (newWorkerId != null)
        team.GetWorkerOrThrow(newWorkerId);

    visit.SetAssignedWorker(newWorkerId);

    var previousWorker = previousWorkerId != null
        ? new VisitWorkerChangedPayload.WorkerInfo(previousWorkerId, team.FindMember(previousWorkerId)?.Name)
        : null;
    var newWorker = newWorkerId != null
        ? new VisitWorkerChangedPayload.WorkerInfo(newWorkerId, team.FindMember(newWorkerId)?.Name)
        : null;

    AddDomainEvent(new VisitWorkerChanged(Id, visitId, previousWorker, newWorker, now));
}
```

| Scenario | `newWorkerId` | `previousWorker` | `newWorker` |
|----------|---------------|-------------------|-------------|
| Assign | `"worker-1"` | `null` | `{ worker-1 }` |
| Reassign | `"worker-2"` | `{ worker-1 }` | `{ worker-2 }` |
| Unassign | `null` | `{ worker-1 }` | `null` |

### 2.3 Add `SetAssignedWorker` to `Visit`

**File:** `Jobs/Jobs.Domain/Models/Visit.cs`

```csharp

    AssignedWorkerId = workerId;
```

## Phase 3: Upsert Flow — Emit Assignment Events

### 3.1 Update `Job.UpdateVisits()` signature

**File:** `Jobs/Jobs.Domain/Models/Job.cs`

Add `Team team` parameter to `UpdateVisits()`. This is the same `Team`
object used in `AssignWorker` and `UnassignWorker` — passed from the
handler.

`UpdateVisits()` uses it for:
- **Validation** — `team.GetWorkerOrThrow(id)` rejects worker IDs not
  in the team or with wrong role
- **Event enrichment** — `team.FindMember(id)?.Name` for `VisitWorkerChanged`

### 3.2 Validate worker IDs and detect assignment changes

Currently `UpdateVisits()` creates/updates visits including
`assignedWorkerId` but only raises `VisitCreated` / `VisitStatusChanged`
events and does not validate workers.

Add: for each visit with a non-null `assignedWorkerId`, validate via
`team.GetWorkerOrThrow()`. For existing visits, detect assignment
changes and raise `VisitWorkerChanged`:

```csharp
// Inside the per-visit update loop in UpdateVisits()

// Validate new worker assignment (checks existence + role)
if (input.AssignedWorkerId != null)
    team.GetWorkerOrThrow(input.AssignedWorkerId);

var previousWorkerId = existingVisit.AssignedWorkerId;

existingVisit.Update(input.DateTime, input.Status, input.AssignedWorkerId);

if (previousWorkerId != existingVisit.AssignedWorkerId)
{
    var previousWorker = previousWorkerId != null
        ? new VisitWorkerChangedPayload.WorkerInfo(previousWorkerId, team.FindMember(previousWorkerId)?.Name)
        : null;
    var newWorker = existingVisit.AssignedWorkerId != null
        ? new VisitWorkerChangedPayload.WorkerInfo(existingVisit.AssignedWorkerId, team.FindMember(existingVisit.AssignedWorkerId)?.Name)
        : null;

    RaiseDomainEvent(new VisitWorkerChanged(Id, existingVisit.Id, previousWorker, newWorker, now));
}
```

All three assignment change scenarios must emit the event:

| Before upsert | After upsert | Event payload |
|---------------|--------------|---------------|
| `null` | `"worker-1"` | `previousWorker: null`, `newWorker: { worker-1 }` |
| `"worker-1"` | `"worker-2"` | `previousWorker: { worker-1 }`, `newWorker: { worker-2 }` |
| `"worker-1"` | `null` | `previousWorker: { worker-1 }`, `newWorker: null` |

The third case (unassign via upsert) happens when a visit's
`assignedWorkerId` is sent as `null` in the upsert payload. This must
also raise `VisitWorkerChanged` — not silently clear the field.

### 3.3 New visits — no separate assignment event

When a new visit is created with a non-null `assignedWorkerId`, the
`VisitCreated` event already includes the worker ID. Do **not** emit
an additional `VisitWorkerChanged` — the assignment is part of the
creation, not a change.

`VisitWorkerChanged` is only raised when an **existing** visit's
`AssignedWorkerId` changes (assign, reassign, or unassign).

## Phase 4: Team Fetch & Injection

The handler builds a `Team` instance and passes it into domain methods.
The aggregate uses it for both validation (existence + role) and name
resolution. One API call per request — no per-worker lookups.

### 4.1 Add `GetTeam` to `IJobWorkerService`

**File:** `Jobs/Jobs.Application/Services/Workers/JobWorkerService.cs`

```csharp
Task<Team> GetTeam(string accountId, CancellationToken ct);
```

Implementation:
1. Call `ITofuAuthApiClient.GetTenantUsersAsync(accountId)` to get
   the full team list
2. Map each user to `TeamMember(userId, contactName ?? userName, role)`
   where role is derived from `TenantRole` (`null` → `Admin`,
   `"Worker"` → `Worker`)
3. Return `new Team(members)`

**Why `GetTenantUsersAsync` instead of `GetWorkerName`:**
- `GetWorkerName` only resolves a display name — it does not verify
  the user is actually a member of the account's team
- `GetTenantUsersAsync` returns the authoritative team list from
  Tofu.Auth, confirming the user has an active `UserTenantRole` record
  and providing the role

### 4.2 Fetch team and pass to domain

**File:** `Jobs/Jobs.Application/Commands/UpsertJobCommandHandler.cs`

The existing upsert flow uses `GetWorkerNames` (batch name resolution)
for visits with `assignedWorkerId`. Replace with a single team fetch:

1. Call `GetTeam(accountId)` once → get `Team` instance
2. Pass `team` to domain methods (`UpdateVisits`, `AssignWorker`,
   `UnassignWorker`)
3. The **domain validates** worker IDs via `team.GetWorkerOrThrow()`
   which checks both existence and role

The handler no longer validates worker IDs itself — it delegates to the
aggregate, which enforces the invariant consistently regardless of the
call path.

### 4.3 Error handling

`Team.GetWorkerOrThrow()` throws `EntityNotFoundException` when the
worker is not found or has the wrong role. This reuses the existing
`NotFound` error code — no dedicated `WorkerNotFound` error needed.

### 4.4 Consider: validate worker not already assigned to conflicting visit

**Open question:** Should the system prevent assigning the same worker to
two visits at the same date/time? If yes:
- Query other visits for the same account + worker + overlapping date
- This is a scheduling concern and may belong in a later stage

**Recommendation:** Skip for now. Flag in overview as future enhancement.
