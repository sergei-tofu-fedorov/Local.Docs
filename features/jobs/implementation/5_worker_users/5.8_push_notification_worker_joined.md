Stage 5.8 - Push Notification When Worker Joins Team
=====================================================

Context
-------

When a worker accepts an invitation and creates their contact info for
the first time, the inviter should receive a push notification. Push
sending lives in Invoices.Backend, which already owns the OnePush
infrastructure and the `INotificationDispatcher` for in-app
notifications.

The notification is **not** sent immediately on acceptance. Instead it
is deferred until the worker creates their contact info for the first
time after accepting the invitation. This ensures:
1. The worker's tenant-specific contact name is available for the push
   message (no fallback to global name or email needed).
2. The notification acts as a signal that the worker has fully joined
   ŌĆö accepted the invite **and** set up their profile.

The push is sent **only to iOS** devices. The product key is always
`ProductConst.FieldService` (`"tofu-fieldservice"`), regardless of
which product the worker used to accept the invitation.

See [PushNotifications.md](../../../Backend/HowTo/PushNotifications.md)
for the full push system architecture.

Trigger: First Contact Info Insert (null ŌåÆ non-null)
-----------------------------------------------------

The push fires when Tofu.Auth detects the first insert of the worker's
tenant-specific contact info (contact info goes from non-existent to
existing). Auth already owns contact storage, so it is the natural
place to detect this transition. No additional state is stored in
Invoices.Backend ŌĆö Auth's `IsCreated` response flag is
the sole trigger.

```
Worker creates contact info for the first time
  ŌåÆ PUT /api/me/contact  (Invoices.Backend)
    ŌåÆ AuthorizationController.UpdateMyContact()
      ŌåÆ _tofuAuthApiClient.UpdateMyContactInfoAsync(...)
        ŌåÆ Auth detects nullŌåÆnon-null contact info, returns { IsCreated: true }
      ŌåÆ if response.IsCreated && masterUser.IsWorkerIn(accountId):
        ŌåÆ _teamNotificationService.SendWorkerJoinedPushAsync(accountId, contactName)
          ŌåÆ load account owner via _masterUserRepository.FindOwnerForAccountId(accountId)
          ŌåÆ check owner has iOS platform (PlatformUserLinks.Any(p => p.Platform == Platform.IOS))
          ŌåÆ build iOS push template via _pushNotificationTemplateService.GetWorkerJoinedTemplate(contactName)
          ŌåÆ _pushService.SendWithParams(accountId, template)
```

### How It Works

1. When a worker accepts an invitation, Auth creates the tenant
   membership with no contact info (existing behaviour). No additional
   state is written in Invoices.Backend.
2. When the worker calls `PUT /api/me/contact` for the first time,
   `UpdateMyContactInfoAsync` in Tofu.Auth detects that contact info
   went from non-existent to existing (insert) and returns
   `IsCreated = true` in the response.
3. `AuthorizationController` checks the response flag. If `true`
   **and** `masterUser.IsWorkerIn(AccountId)`, delegates to
   `ITeamNotificationService.SendWorkerJoinedPushAsync(accountId, contactName)`.
   The service:
   a. Loads the account owner via
      `_masterUserRepository.FindOwnerForAccountId(accountId)`.
   b. Checks the owner has an iOS platform link
      (`owner.PlatformUserLinks.Any(p => p.Platform == Platform.IOS)`).
   c. Builds the push template via
      `_pushNotificationTemplateService.GetWorkerJoinedTemplate(contactName)`
      and sends it via `IPushService`.
4. Push failure is caught and logged ŌĆö it must not break the contact
   update flow. No retry mechanism; the push is best-effort.

### Getting the Worker Contact Name

The push is sent after the worker updates contacts, so the
tenant-specific contact name is available from the contact update
payload itself ŌĆö no extra Auth call needed.

Data Available for Push
------------------------

At the time of the contact update (when the push fires):

| Source | Field | Maps To Push Field |
|--------|-------|--------------------|
| Contact update payload | `Name` (required) | `contactName` (tenant-specific) |
| Request headers | `AccountId` | `accountId` for `SendWithParams` |
| Hardcoded | `ProductConst.FieldService` | `productKey` for `PushNotificationTemplate` |
| Account owner lookup | `PlatformUserLinks[].Platform` | iOS platform gate |

Push Notification Content
-------------------------

Uses the existing `InvoicesAlert` OnePush template. Since the push
targets iOS only, it uses the iOS format:

- **iOS**: `{ message: "{JSON}", type, push_id }` ŌĆö `message` is a
  serialized JSON string with `title` and `body` fields

Push message fields:

| Field | Value |
|-------|-------|
| **title** | `"{contactName} has joined your team"` |
| **body** | `"You can now assign {contactName} to jobs"` |
| **type** | `"automatic"` |
| **push_id** | `Consts.EventPushTypes.InvitationAccepted` |

### Step 3. Add `ITeamNotificationService`

Extract push logic into a dedicated notification service, following the
`PaymentsNotificationService` pattern.

**Interface:** `Src/Invoices.Common/Services/Push/ITeamNotificationService.cs`

```csharp
namespace Invoices.Common.Services.Push;

/// <summary> Push notifications for team membership events. </summary>
public interface ITeamNotificationService
{
    Task SendWorkerJoinedPushAsync(string accountId, string? contactName, CancellationToken ct);
}
```

**Implementation:** `Src/Invoices.Api/Services/TeamNotificationService.cs`

The implementation lives in `Invoices.Api/Services/` (alongside `TeamService`)
because it depends on `IPushNotificationTemplateService` from `Tofu.Email`,
which is only referenced by the API project.

```csharp
namespace Invoices.Api.Services;

/// <summary> Push notifications for team membership events. </summary>
public class TeamNotificationService : ITeamNotificationService
{
    private readonly IPushService _pushService;
    private readonly IPushNotificationTemplateService _pushNotificationTemplateService;
    private readonly IMasterUserRepository _masterUserRepository;
    private readonly ILogger<TeamNotificationService> _logger;

    public async Task SendWorkerJoinedPushAsync(
        string accountId, string? contactName, CancellationToken ct)
    {
        try
        {
            if (string.IsNullOrWhiteSpace(contactName))
                throw new ArgumentException("Contact name is required", nameof(contactName));

            var owner = await _masterUserRepository
                .FindOwnerForAccountId(accountId, ct);

            if (owner?.PlatformUserLinks.Any(p => p.Platform == Platform.IOS) != true)
                return;

            var template = _pushNotificationTemplateService
                .GetWorkerJoinedTemplate(contactName);

            await _pushService.SendWithParams(accountId, template);
        }
        catch (Exception ex)
        {
            _logger.LogError(ex,
                "Failed to send worker joined push for account");
        }
    }
}
```

The `contactName` is validated up front ŌĆö if null or whitespace, an
`ArgumentException` is thrown and caught by the outer handler, so the
push is silently skipped and the error logged. This avoids sending a
push with a blank name.

### Step 4. Modify AuthorizationController (contact update)

**File:** `Src/Invoices.Api/Controllers/AuthorizationController.cs`

Replace `IPushService` + `IPushNotificationTemplateService` +
`IMasterUserRepository` with a single `ITeamNotificationService`
dependency. After calling Auth API, check the `IsCreated` flag and
Worker role, then delegate to the service:

```csharp
[HttpPut("contact")]
public async Task<IActionResult> UpdateMyContact(
    [FromBody] UpdateTeamMemberContactRequestDto request,
    CancellationToken ct)
{
    var masterUser = GetRequiredMasterUser();

    var response = await _tofuAuthApiClient.UpdateMyContactInfoAsync(
        AccountId, request.ToUpdateUserContactInfoRequest(), ct);

    if (response.IsCreated
        && masterUser.IsWorkerIn(AccountId))
    {
        await _teamNotificationService.SendWorkerJoinedPushAsync(
            AccountId, request.Name, ct);
    }

    return NoContent();
}
```

Trade-offs
----------

- **iOS only**: Push is sent only when the account owner uses an iOS
  device. Android users do not receive this notification.
- **Coupling**: Push logic is in Invoices.Backend, not in Auth. If
  another client accepts invitations directly via Auth API (bypassing
  Invoices), no push is sent.
- **No retry**: If the push fails after a successful contact info
  creation, contact info already exists ŌĆö the push won't re-trigger.
  This is acceptable because the notification is best-effort, not a
  critical business operation. The worker has still joined
  successfully.
- **Auth contract change**: Requires updating `Tofu.Auth.Api.Client`
  NuGet package. The `UpdateMyContactInfoAsync` return type changes
  from `Task` to `Task<UpdateContactInfoResponse>`. This is a
  breaking change to the client contract but only affects
  Invoices.Backend (the sole consumer).
- **No duplicate protection**: If a worker's contact info is somehow
  deleted and re-created, the push fires again. This is unlikely in
  practice and harmless ŌĆö a duplicate "X joined your team" push is
  not confusing.
- **Two-step flow**: If a worker accepts but never updates contacts,
  the inviter is never notified. This is intentional ŌĆö the
  notification signals that the worker has fully joined.
- **Extra query for owner**: When a worker creates contact info,
  Invoices.Backend loads the account owner's `MasterUser` to check
  their platform. The worker role check uses `GetRequiredMasterUser()`
  (already loaded by middleware), so the extra query only happens for
  confirmed workers on first contact creation ŌĆö a rare event.
