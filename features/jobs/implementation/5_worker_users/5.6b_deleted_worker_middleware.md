# Step 5.6b: Deleted Worker — Middleware Guard for Exempt Endpoints

> After [5.6a](5.6a_deleted_worker_forbidden.md) introduces the 403 flow,
> signature-exempt endpoints (`/api/email`, `/api/logo`, `[IgnoreSignature]`)
> will also be blocked. This step restores their fallthrough behavior.

## Problem

With 5.6a applied, `AccountAccessDeniedException` propagates through the
wrapper and hits the global exception handler for **all** endpoints.

Signature-exempt endpoints are server-to-server callbacks that work without
bearer tokens or signatures. If a request happens to include a bearer token
with a mismatched Account-Id, they should still fall through — not 403.

### Signature-exempt endpoints (must keep working)

| Endpoint | Why exempt |
|----------|-----------|
| `/api/email/*` | Email webhook callbacks |
| `/api/logo/*` | Logo upload/serving |
| `[IgnoreSignature]` endpoints: | Payment provider callbacks |
| `GET /callback/payments/auth/{providertype}` | Stripe/PayPal OAuth return |
| `GET /callback/payments/auth/{providerType}/refresh-link` | Token refresh callback |
| `GET /callback/payments/success/{providertype}` | Payment success redirect |
| `GET /callback/payments/fail/{providertype}` | Payment failure redirect |

---

## Solution

### Middleware catch with `when (IsSignatureExempt)`

**File:** `Invoices.Api/Middleware/AccountAuthenticationMiddleware.cs`

Wrap the auth call in a try-catch that only catches for exempt endpoints:

```csharp
AuthenticationInfo? authenticationInfo;
try
{
    authenticationInfo = await authApiAuthenticationService.AuthenticateWithAuthApi(new AuthContext
    {
        Platform = platform.GetPlatform(),
        ProductKey = productKey ?? "unknown",
        ExplicitlySpecifiedAccountId = accountId,
    }, _logger, CancellationToken.None);
}
catch (AccountAccessDeniedException) when (IsSignatureExempt(requestUrl, endpoint))
{
    authenticationInfo = null;
}
```

The `when` filter means:
- **Normal endpoint** -> exception not caught -> propagates -> 403
- **Signature-exempt endpoint** -> caught, set null, fall through (existing behavior)

### Extract `IsSignatureExempt` helper

Same conditions as the existing signature check block, extracted for reuse:

```csharp
private static bool IsSignatureExempt(string? requestUrl, Endpoint? endpoint)
{
    return requestUrl?.Contains("/api/email") == true
        || requestUrl?.Contains("/api/logo") == true
        || endpoint == null
        || endpoint.Metadata.GetMetadata<IgnoreSignatureAttribute>() != null;
}
```

Reuse in the existing signature check:

```csharp
if (!IsSignatureExempt(requestUrl, endpoint))
{
    // validate signature...
}
```

---

## Result

### Normal endpoints (e.g. `/api/jobs`)

```
Bearer -> account mismatch -> AccountAccessDeniedException -> re-thrown by wrapper -> 403
```

### Signature-exempt endpoints (`/api/email`, `/api/logo`, `[IgnoreSignature]`)

```
Bearer -> account mismatch -> re-thrown by wrapper -> caught by middleware (when exempt) -> same fallthrough
```

No behavior change for exempt endpoints. Security hole closed for all others.
