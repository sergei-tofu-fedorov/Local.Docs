# Step 5.5: Admin Contact Resolution

## Problem

The Admin user in a team is the owner of the business (Account). When the system needs the Admin's contact details (email, phone) — e.g. for notifications, job timeline, or worker-facing views — it currently resolves them from the Auth service (`GetTenantUsersAsync`).

However, the account owner may have configured different business contacts on the Account itself (Account Contacts). These should take priority over the personal contacts stored in Auth, because the business-facing contact info is what workers and clients should see.

## Contact Sources

| Source | Model | Fields | Storage |
|--------|-------|--------|---------|
| Account Contacts | `AccountContacts` | Name, Phone, Email, Address | `Account.Contacts` in MongoDB |
| Auth Service | `TenantUserResponseDto` | ContactName, Email, ContactPhoneNumber | Auth API via `GetTenantUsersAsync` |

## Solution

When resolving contacts for the Admin team member, use **Account Contacts** as the primary source with a **fallback to Auth service** contacts.

Resolution rule per field (applies only to Admin role members):

```
resolved.Email = AccountContacts.Email ?? AuthUser.Email
resolved.Name  = AccountContacts.Name  ?? AuthUser.ContactName ?? AuthUser.UserName
resolved.Phone = AccountContacts.Phone ?? AuthUser.ContactPhoneNumber
```

Worker role members are unaffected — they always use Auth service contacts.

### Existing Precedent

`EmailService` already implements this pattern for email:

```csharp
// EmailService.cs — Account.Contacts?.Email first, MasterUser fallback
var email = !string.IsNullOrWhiteSpace(accountInfo.Email)
    ? accountInfo.Email
    : !string.IsNullOrWhiteSpace(link.OriginalEmail)
        ? link.OriginalEmail
        : throw new InvalidOperationException(...);
```

## Changes

### 1. Introduce `TeamService` — `Invoices.Api/Services/TeamService.cs`

Create a new `TeamService` in `Invoices.Api` that fully encapsulates `ITofuAuthApiClient` for all team operations. `TeamController` depends only on `ITeamService` — no direct Auth client usage.

**Interface** (`ITeamService`):

```csharp
public interface ITeamService
{
    Task<TeamMemberListResponseDto> GetTeamMembers(string accountId, Guid currentUserId, CancellationToken ct);
    Task<TeamMemberDto> GetTeamMember(string accountId, Guid userId, Guid currentUserId, CancellationToken ct);
    Task UpdateMemberContact(string accountId, Guid userId, UpdateTeamMemberContactRequestDto request, CancellationToken ct);
    Task RemoveMember(string accountId, Guid userId, string? requestedByMasterUserId, CancellationToken ct);
}
```

**Implementation** (`TeamService`):

- Depends on `ITofuAuthApiClient`, `IAccountsRepository`, `IMasterUserRepository`, `IHandlerDispatcher`.
- `currentUserId` is passed by the controller from `AuthenticationInfo.MasterUserId` (resolved by middleware) — avoids an extra `GetAuthenticatedUserInfoAsync` call.
- Loads `Account.Contacts` lazily: `GetTeamMember` skips the account query for non-Admin users; `GetTeamMembers` always loads contacts (multiple members may include an admin).
- Admin contact overlay is handled by a private `ApplyAdminContacts` method (keeps `Mapping.cs` simple).
- `RemoveMember` orchestrates visit unassignment, Auth revocation, and local record cleanup.

```csharp
public class TeamService : ITeamService
{
    private readonly ITofuAuthApiClient _authClient;
    private readonly IAccountsRepository _accountsRepository;
    private readonly IMasterUserRepository _masterUserRepository;
    private readonly IHandlerDispatcher _dispatcher;

    public async Task<TeamMemberListResponseDto> GetTeamMembers(
        string accountId, Guid currentUserId, CancellationToken ct)
    {
        var users = await _authClient.GetTenantUsersAsync(accountId, ct);
        var contacts = (await _accountsRepository.GetAsync(accountId)).Contacts;

        return new TeamMemberListResponseDto
        {
            TeamMembers = users
                .Select(u => ApplyAdminContacts(u.ToTeamMemberResponseDto(currentUserId), u.Role.Level, contacts))
                .ToList()
        };
    }

    public async Task<TeamMemberDto> GetTeamMember(
        string accountId, Guid userId, Guid currentUserId, CancellationToken ct)
    {
        var user = await _authClient.GetTenantUserAsync(accountId, userId, ct);
        var dto = user.ToTeamMemberResponseDto(currentUserId);

        if (user.Role.Level != AuthRoleLevel.Admin)
            return dto;

        var contacts = (await _accountsRepository.GetAsync(accountId)).Contacts;
        return ApplyAdminContacts(dto, user.Role.Level, contacts);
    }

    public async Task RemoveMember(
        string accountId, Guid userId, string? requestedByMasterUserId, CancellationToken ct)
    {
        var memberMasterUserId = userId.ToString().ToLowerInvariant();

        // 1. Unassign all visits before revoking access
        var command = new UnassignWorkerFromAllVisitsCommand(accountId, memberMasterUserId, requestedByMasterUserId);
        await _dispatcher.DispatchCommand<...>(command, ct);

        // 2. Revoke Auth access
        await _authClient.RemoveUserFromTenantAsync(accountId, userId, ct);

        // 3. Remove local record
        await _masterUserRepository.RemoveMemberFromAccount(memberMasterUserId, accountId, ct);
    }

    private static TeamMemberDto ApplyAdminContacts(
        TeamMemberDto dto, AuthRoleLevel roleLevel, AccountContacts? contacts) { ... }
}
```

Register via DI (Scrutor will pick it up, or add explicit registration in `DIConfig`).

### 2. Team Member Mapping — `Invoices.Api/Dto/Team/Mapping.cs`

`Mapping.cs` handles basic DTO conversion only (`ToTeamMemberResponseDto(Guid currentUserId)`). The admin contact overlay is applied separately by `TeamService.ApplyAdminContacts` — this keeps the mapping pure and the overlay logic co-located with the account query that provides the contacts.

### 3. TeamController — `Invoices.Api/Controllers/TeamController.cs`

The controller depends only on `ITeamService`. It passes `Guid.Parse(AuthenticationInfo!.MasterUserId)` for the `currentUserId` parameter (middleware already resolves this — no extra API call needed).

```csharp
public class TeamController : BaseController
{
    private readonly ITeamService _teamService;

    // GetTeamMembers          → _teamService.GetTeamMembers(AccountId, currentUserId, ct)
    // GetTeamMember           → _teamService.GetTeamMember(AccountId, userId, currentUserId, ct)
    // UpdateTeamMemberContact → _teamService.UpdateMemberContact(AccountId, userId, request, ct)
    // RemoveTeamMember        → _teamService.RemoveMember(AccountId, userId, masterUserId, ct)
}
```

### 4. Jobs Domain — No Admin Contacts Needed

`JobWorkerService.GetTeam()` resolves team member names for job/visit context. Currently it uses `ContactName ?? UserName` from the Auth service for all members, including admins. This is sufficient because Jobs only display the team member's name — not email or phone — and the Auth service name is adequate for this purpose.

## Out of Scope

Worker-facing DTOs (`WorkerClientResponse`, `WorkerVisitDetailsResponse`) deliberately exclude admin contact info — they only show client address. This is correct by design and unaffected by this step.
