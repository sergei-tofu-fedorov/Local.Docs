# Step 5.6a: Deleted Worker — Forbidden Exception Flow

> Replace `InvalidOperationException` with a proper 403 Forbidden response when
> bearer-authenticated users request an account they don't belong to, while
> preserving backward compatibility with signature-based auth.

## Problem

When `AuthApiAuthenticationService.AuthenticateWithAuthApi` throws
`InvalidOperationException` (user authenticated but Account-Id doesn't
match any of their accounts), the extension wrapper in
`ServiceProviderExtensions.cs` swallows it:

```csharp
catch (AuthenticationInfoMissedException) { return null; }  // no token -> try signature
catch (ApiAuthenticationFlowException)    { throw; }        // known auth error -> propagate
catch (Exception e)                       { return null; }  // <- swallows account mismatch
```

The middleware receives `null` and falls through to the signature path,
causing either wrong-account access or misleading "Invalid signature" errors.

All three switch arms are affected:

```csharp
var accountId = allAccountIds.Count switch
{
    0 => ... throw new InvalidOperationException("User don't own any account..."),
    1 => ... throw new InvalidOperationException("User owns single account..."),
    _ => ... throw new InvalidOperationException("User owns accounts [...]...")
};
```

## Solution

### Design: Separate authN and authZ exception hierarchies + deferred fallback

The existing `ApiAuthenticationFlowException` covers **authentication** errors
(token expired, token revoked, deleted master, missing claims). Account
mismatch is an **authorization** error — the user is who they say they are,
they just don't have access to the requested account.

New hierarchy (authZ):

```
ApiAuthorizationException               <- "you are who you say, but can't do this"
+-- AccountAccessDeniedException        <- user has no access to the requested account
```

**Backward-compatible fallback:** When bearer auth fails with an authorization
error, the middleware defers the exception and tries signature auth. If
signature succeeds, the request proceeds normally. If signature also fails, the
original authorization exception is re-thrown (with preserved stack trace via
`ExceptionDispatchInfo`).

```
Bearer auth
  ├─ Success → proceed as AuthenticationApi
  ├─ ApiAuthorizationException → defer, try signature auth
  │     ├─ No accountId → throw deferred exception
  │     ├─ Signature succeeds → proceed as AccountIdWithSignature
  │     ├─ Signature fails → throw deferred exception (original stack trace)
  │     └─ Signature not required → proceed as AccountIdWithSignature
  ├─ ApiAuthenticationFlowException → propagate immediately (token expired, etc.)
  └─ Other error / no token → return null → signature fallback (existing behavior)
```

This approach replaces the originally planned [5.6b middleware guard](5.6b_deleted_worker_middleware.md)
— signature-exempt endpoints are handled naturally because the fallback
always tries signature auth before giving up.

## Phase 1: New Exception Hierarchy

### 1.1 Create `ApiAuthorizationException` base class

**File:** `Invoices.Core/Exceptions/ApiAuthorizationException.cs` (new)

```csharp
namespace Invoices.Core.Exceptions;

/// <summary>
/// Base exception for authorization errors that should be propagated to the caller.
/// User is authenticated, but does not have access to the requested resource.
/// </summary>
public abstract class ApiAuthorizationException : Exception
{
    protected ApiAuthorizationException(string message) : base(message) { }
    protected ApiAuthorizationException(string message, Exception innerException)
        : base(message, innerException) { }
}
```

### 1.2 Create `AccountAccessDeniedException`

**File:** `Invoices.Core/Exceptions/AccountAccessDeniedException.cs` (new)

```csharp
namespace Invoices.Core.Exceptions;

/// <summary> User is authenticated but has no access to the requested account. </summary>
public sealed class AccountAccessDeniedException : ApiAuthorizationException
{
    public string UserId { get; }
    public string RequestedAccountId { get; }

    public AccountAccessDeniedException(string userId, string requestedAccountId)
        : base($"User '{userId}' has no access to account '{requestedAccountId}'")
    {
        UserId = userId;
        RequestedAccountId = requestedAccountId;
    }
}
```

## Phase 5: Middleware — Deferred Authorization Fallback

### 5.1 Catch `ApiAuthorizationException` and defer to signature auth

**File:** `Invoices.Api/Middleware/AccountAuthenticationMiddleware.cs`

The middleware catches `ApiAuthorizationException` from bearer auth, captures
it with `ExceptionDispatchInfo`, then falls through to signature verification:

```csharp
var (authenticationInfo, deferredAuthorizationException) = await TryAuthenticateWithBearer(
    authApiAuthenticationService, platform.GetPlatform(), productKey, accountId);
```

`TryAuthenticateWithBearer` returns `(AuthenticationInfo?, ExceptionDispatchInfo?)`:
- On success → `(info, null)`
- On `ApiAuthorizationException` → `(null, ExceptionDispatchInfo.Capture(ex))`

### 5.2 Re-throw deferred exception when signature also fails

If no accountId is present:

```csharp
if (string.IsNullOrEmpty(accountId))
{
    deferredAuthorizationException?.Throw();
    throw new AuthenticationException("Account-Id header is not present.");
}
```

If signature verification is required and fails:

```csharp
try
{
    await VerifySignature(headers, auth, accountId, context.Request);
}
catch (AuthenticationException) when (deferredAuthorizationException != null)
{
    deferredAuthorizationException.Throw();
}
```

`ExceptionDispatchInfo.Throw()` preserves the original stack trace from
bearer auth.

### 5.3 Signature not required — proceed normally

When signature verification is not required (email/logo/`[IgnoreSignature]`
endpoints), the middleware skips verification and proceeds as
`AccountIdWithSignature`. The deferred exception is not thrown — this
preserves backward compatibility for server-to-server callbacks.

---

## Result

### Before

```
Bearer -> account mismatch -> InvalidOperationException -> swallowed -> signature fallback -> wrong access or bad error
```

### After — signature fails or missing

```
Bearer -> account mismatch -> ApiAuthorizationException -> deferred -> signature fails -> 403 Forbidden (original stack trace)
```

### After — signature succeeds (backward compatibility)

```
Bearer -> account mismatch -> ApiAuthorizationException -> deferred -> signature OK -> proceed as AccountIdWithSignature
```

### After — signature-exempt endpoint

```
Bearer -> account mismatch -> ApiAuthorizationException -> deferred -> signature not required -> proceed as AccountIdWithSignature
```

## iOS

> See also: [iOS Backend Error Handling](../../../../IOS/backend_error_handling.md)

Introducing a new error code (e.g. replacing `"forbidden"` with something more specific) will **not break** the iOS app. The `forbidden` case is defined in the `ApiErrorType` enum but is never checked or matched anywhere in the codebase. Any unrecognized error code falls through to `.unknownError` and the app simply displays the backend's `message` field as-is. The HTTP status code is irrelevant to this flow — iOS only looks at `errorCode` and `message` from the JSON response body.

---

## WEB

> See also: [Web Backend Error Handling](../../../../Web/backend_error_handling.md)

Introducing a new error code instead of `"forbidden"` will **not break** the web frontend. The `forbidden` type is defined in the `APIErrorType` union (`src/shared/lib/http/types.ts`) but is never explicitly checked or handled by any feature code. Error parsing in `checkErrorByContent()` (`src/shared/lib/http/errors.ts`) reads the `code` field from the response body and assigns it to `error.type` — TypeScript unions aren't enforced at runtime, so an unrecognized code simply becomes a string value. The generic error flow still works: the backend's `message` is displayed in a toast notification via `showErrorNotificationFx`, and the error is reported to Sentry/Amplitude as usual. No frontend changes are required.
