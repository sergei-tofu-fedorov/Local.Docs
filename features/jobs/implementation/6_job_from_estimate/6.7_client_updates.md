# Step 6.7: Client Application Updates

Summary of all contract and behavioral changes that affect client applications (web, mobile) in the Step 6 implementation. Clients must adopt these changes to work correctly with the new job-from-estimate flow.

## 1. Estimate Done status — client responsibility

**The backend does NOT transition an estimate to Done after job creation.** When the client creates a job from an estimate (`POST /api/v3/jobs/from-estimate`), the backend:
1. Validates the estimate has **Approved** status (rejects all other statuses).
2. Creates the job and links it to the estimate server-side.

After receiving the successful response, **the client must make a separate call to update the estimate status to Done** (e.g. `PUT /estimates` with `status: Done`). This is required because:
- The Done status signals to other parts of the UI that the estimate has been converted.
- Other clients use the Done status to disable the "create invoice from estimate" button — an estimate linked to a job that already has an invoice should be in Done status.
- The backend relies on the client to manage status transitions for estimates (the server only enforces the Approved gate on job creation).

**Why not automatic?** Status transitions on estimates are client-driven. The backend does not know when the client has finished its local UI operations (e.g. showing a success screen, navigating to the job). Making the status change explicit gives the client full control over timing.

## 2. Deprecated fields on job upsert — migrate to invoice/estimate saves

`InvoiceId` and `EstimateId` on the job upsert request (`PUT /jobs`) are now **deprecated**. They still work for backward compatibility but will be removed in a future update (6.6).

### Old flow (deprecated)

```
PUT /jobs { clientId, invoiceId: "inv-1", estimateId: "est-1", ... }
```

The job received invoice/estimate IDs directly from the client on upsert. The backend resolved the links via `FetchAndApplyRelations`.

### New flow (preferred)

```
PUT /invoices  { jobId: "job-1", estimateId: "est-1", ... }
PUT /estimates { jobId: "job-1", ... }
```

The client sets `JobId` on invoices and estimates when saving them. The backend automatically propagates the link to `Job.Relations` through internal sync mechanisms (`TryUpdateJobSummary`, `TryUpdateJobLink`). The job never receives relation IDs from client input — its relations are derived server-side.

### Migration steps for clients

1. Stop sending `InvoiceId`/`EstimateId` on `PUT /jobs` requests.
2. When creating an invoice for a job, include `jobId` (and optionally `estimateId`) on the `PUT /invoices` request.
3. When linking an estimate to a job, include `jobId` on the `PUT /estimates` request.
4. Read `InvoiceId`/`EstimateId` from the job response as before (these are still returned, derived from `Job.Relations`).

## 3. New `EstimateId` field on Invoice

Invoices now have an optional `EstimateId` field. The client should send this when creating an invoice from an estimate.

**Request:** `PUT /invoices { ..., estimateId: "est-1", jobId: "job-1" }`

**Why:** This enables the backend to:
- Resolve `JobId` from the estimate if the client doesn't provide it.
- Maintain the full Estimate ↔ Invoice ↔ Job relation chain.
- Cross-link entities automatically: when saving an invoice with a `JobId`, the backend looks up the job and, if it has a linked estimate, sets `EstimateId` on the invoice. Conversely, when saving an estimate with a `JobId`, if the job has an invoice, the backend sets `InvoiceId` on the estimate.

**Response:** The invoice response includes `estimateId` if set.

## 4. New `InvoiceId` field on Estimate

Estimates now have an optional `InvoiceId` field. This is set **server-side** (not by the client) when the backend detects a linked invoice through the shared job.

**How it works:**
- Client saves an estimate with `JobId`.
- Backend looks up the job and finds it already has an invoice.
- Backend sets `estimate.InvoiceId = job.InvoiceId` before saving.

The client can read `InvoiceId` from estimate responses but does not need to send it.

## 5. New `CurrencyCode` on Job response

Job responses now include a `CurrencyCode` field.

- When a job is created from an estimate, the job inherits the estimate's currency.
- Default is `USD` if no currency is set.
- The display currency falls back to `Job.CurrencyCode` instead of a static constant, so a job created from a EUR estimate shows EUR even before an invoice is linked.

**Response field:** `currencyCode` (string, nullable) in `JobDto`.

## 6. Duplicate link prevention — server-side validation

The backend now validates that a job does not already have a different invoice or estimate linked. These checks run inside the `Add` call (not on the job upsert).

### Invoice save (`PUT /invoices`)

If the invoice has a `JobId` and the job already has a **different** invoice linked:
- Returns `400 Bad Request` with error code `jobAlreadyHasInvoice`.
- Updating the **same** invoice (same ID) passes validation.

### Estimate save (`PUT /estimates`)

If the estimate has a `JobId` and the job already has a **different** estimate linked:
- Returns `400 Bad Request` with error code `jobAlreadyHasEstimate`.
- Updating the **same** estimate (same ID) passes validation.

### Client-side recommendations

- Use the estimate **Done** status to hide the "create invoice" button when the estimate has already been converted.
- Handle `jobAlreadyHasInvoice` and `jobAlreadyHasEstimate` error responses gracefully (e.g. show a message that the job already has a linked document).

## 7. Sync scenarios — graceful handling of unsynced jobs

When the client saves an invoice or estimate with a `JobId` that hasn't been synced to the backend yet:
- The save proceeds normally. The `JobId` is persisted on the entity.
- `Job.Relations` will be populated once the job arrives and subsequent saves trigger the sync.
- Timeline events are created with an empty `jobNumber`. The client should handle missing job numbers in display text (e.g. show "Job" instead of "Job #123").

No special action is required from the client — the backend handles missing jobs gracefully.

## 8. Client lookup changes (internal, no API impact)

The backend now uses `TryGetClient` (returns `null`) instead of `GetClient` (throws) when creating or updating a job. This means:
- A job can be created even if the client (customer) hasn't been synced to the catalog yet, as long as a `Title` is provided.
- If the client is not found, the client snapshot is not stored on the job. It will be populated on the next upsert when the client becomes available.

**Client impact:** None directly. However, clients should ensure they send a `Title` on job creation when the client catalog entry might not exist yet.

## 9. Onboarding updates

### `isFromGenerator` flag

The onboarding status response now includes `isFromGenerator` (boolean). Clients use this to decide whether to show the job suggestion modal — the modal is only relevant for non-generator users.

### Modal dismissal counter

A new generic modal dismissal mechanism is available:

**Dismiss endpoint:** `POST /onboarding/dismiss-modal`
```json
{ "modal": "EstimateToJob" }
```

**Response:** Full `OnboardingStatus` including `modalDismissals`:
```json
{
  "modalDismissals": [
    { "modalType": "EstimateToJob", "dismissCount": 2 }
  ]
}
```

**Client behavior:**
1. When a user tries to convert an estimate directly to an invoice, show a modal suggesting they create a **job** instead.
2. If the user dismisses the modal, call `POST /onboarding/dismiss-modal` with `modal: "EstimateToJob"`.
3. Read `dismissCount` from the response. When it reaches 3 (or the client's threshold), stop showing the modal.
4. On app startup, read `modalDismissals` from the onboarding status to know whether to show the modal.

## 10. Timeline events

New timeline events are generated by the backend:

| Event | Where it appears | Payload | When generated |
|-------|-----------------|---------|----------------|
| `jobCreated` | Job timeline | `{ }` | Any new job (manual or from estimate) |
| `estimateJobCreated` | Estimate timeline | `{ "jobId", "jobNumber", "docNumber" }` | Estimate's `JobId` changes from null to non-null (via `LinkJobToEstimate` RPC or estimate save with `JobId`) |
| `estimateCreated` | Estimate timeline | `{ "docNumber" }` | New standalone estimate (no `JobId`) |
| `estimateCreatedFromJob` | Estimate timeline | `{ "jobId", "jobNumber", "docNumber" }` | New estimate created with a `JobId` |
| `invoiceCreatedFromJob` | Invoice timeline | `{ "jobId", "jobNumber" }` | New invoice saved with a `JobId` |

**Display considerations:**
- `jobNumber` may be empty in sync scenarios (job not yet synced). The client should display a fallback (e.g. "Job" without a number).
- `estimateJobCreated` includes `docNumber` (the estimate's document number) for cross-reference.

**Backward compatibility for estimate creation events:**
- Old estimates created before this change have an `estimateStatusChanged` event with payload `{ from: null, to: "draft" }` as their creation event.
- When no `estimateCreated` or `estimateCreatedFromJob` event exists for an estimate, clients should treat `estimateStatusChanged(null → draft)` as the creation event.

**Obsolete events:**

The following events are no longer emitted by the backend. They remain in old timelines but clients should not expect them for new entities.

| Obsolete event | Replaced by | Reason |
|----------------|-------------|--------|
| `jobCreatedFromEstimate` | `estimateJobCreated` | Job creation from estimate now appears on the estimate timeline, not the job timeline |
| `estimateStatusChanged` (`null → draft`) | `estimateCreated` / `estimateCreatedFromJob` | Dedicated creation events replace the generic status-change piggyback |
| `estimateStatusChanged` (`approved → done`) | `estimateInvoiceCreated` | Invoice creation from estimate now has a dedicated event |

## 11. New `source` field on Invoice and Estimate responses

Invoice and estimate responses now include a `source` string field that indicates how the entity was originally created.

### Invoice `source` values

| Value | Meaning |
|-------|---------|
| `"unknown"` | Legacy record created before this field existed |
| `"none"` | Created standalone (no job or estimate) |
| `"estimate"` | Created from an estimate |
| `"job"` | Created from a job |

### Estimate `source` values

| Value | Meaning |
|-------|---------|
| `"unknown"` | Legacy record created before this field existed |
| `"none"` | Created standalone (no job) |
| `"job"` | Created from a job |

**Key behaviors:**
- Source is set once at creation time and never changes, even if the entity is later linked to a job or estimate.
- The client does **not** send `source` — it is server-computed and read-only.
- Legacy records that existed before this change return `"unknown"`. Clients should treat this as "source not available" and fall back to inferring from `jobId`/`estimateId` presence if needed.

## Summary of required client changes

| Change | Priority | Action |
|--------|----------|--------|
| Update estimate to Done after job creation | **Required** | Call `PUT /estimates` with Done status after `POST /jobs/from-estimate` |
| Send `EstimateId` on invoice creation from estimate | **Required** | Include `estimateId` in `PUT /invoices` request |
| Send `JobId` on invoice/estimate saves | **Required** | Include `jobId` when saving invoices/estimates linked to a job |
| Stop sending `InvoiceId`/`EstimateId` on job upsert | **Recommended** | Remove deprecated fields from `PUT /jobs` requests |
| Handle `jobAlreadyHasInvoice` / `jobAlreadyHasEstimate` errors | **Required** | Show user-friendly error when duplicate link is attempted |
| Handle empty `jobNumber` in timeline events | **Required** | Display fallback text when job number is missing |
| Handle new estimate creation events (`estimateCreated`, `estimateCreatedFromJob`) | **Required** | Display creation events in estimate timeline; treat `statusChanged(null→draft)` as fallback for old estimates |
| Read `source` from invoice/estimate responses | **Optional** | Use `source` field instead of inferring from `jobId`/`estimateId` presence |
| Read `CurrencyCode` from job response | **Optional** | Use `currencyCode` for display if available |
| Implement "Estimate to Job" suggestion modal | **Recommended** | Use `isFromGenerator` + `modalDismissals` from onboarding |
| Provide `Title` on job creation when client may not exist | **Recommended** | Ensures job creation succeeds even without synced client catalog |
