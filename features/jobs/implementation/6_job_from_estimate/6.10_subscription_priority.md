# Step 6.10: Subscription Tier Priority

**Task**: [FS-795](https://app.clickup.com/t/869cefbcd) — subtask of [FS-756 (BE: Estimate to Job)](https://app.clickup.com/t/869c9f5m9)

## Overview

When a user has multiple active subscriptions (e.g. Invoice Maker + Field Service), the system must select the highest-priority one as "current". Today the selection ignores subscription tier and picks by latest expiration date, which can cause a lower-tier plan (Plus) to win over a higher-tier plan (FsmSolo) if it expires later. This breaks the Estimate-to-Job feature because users with an FS subscription may incorrectly appear as IM-only users and lose access to Jobs.

## Context

The Estimate-to-Job feature ([overview](overview.md)) gates functionality based on the user's current plan:

- **FS subscription (Solo/Team/Business)**: user sees "Create Job first" modal, can convert estimates to jobs
- **IM subscription (Plus/Premium/Invoicing)**: user sees "Convert to Invoice" only, no Jobs access
- **Starter plan** (3 free jobs): user sees "Create Job first" modal

If a user has both an IM and an FS subscription and the system picks the IM one as current, the user loses Jobs access despite having paid for it.

### How users end up with two subscriptions

- **No-auth iOS purchases**: user installs FS app without auth, buys subscription, already had IM subscription
- **Cross-platform purchases**: user has Stripe (web) subscription, then buys via Apple/Google on mobile
- **Account merges**: anonymous-to-authenticated user flow merges subscription histories

## Current State

### Subscription selection: `GetPrimarySubscription`

**File**: [`AccountSubscriptionExtensions.cs`](https://github.com/m-unicorn/Invoices.Backend/blob/master/Src/Invoices.Implementation.Services/Subscription/AccountSubscriptionExtensions.cs)

```csharp
public static AccountSubscription? GetPrimarySubscription(
    this IReadOnlyCollection<AccountSubscription> subscriptions)
{
    return subscriptions.Where(s => s.IsActive).MaxBy(s => s.ExpirationTime) ??
           subscriptions.MaxBy(t => t.StartTime);
}
```

Logic: among active subscriptions, pick the one with the latest `ExpirationTime`. Tier is ignored.

## Changes

### 1. Add `ProductTypePriority` helper to `Invoices.Core`

Create a new static helper class in `Invoices.Core/Models/` that defines the canonical tier ordering. This keeps the priority definition close to the `ProductType` enum and makes it available to all layers without cross-project dependencies.

**New file**: `Invoices.Core/Models/ProductTypePriority.cs`

    {
        return productType switch

            ProductType.FsmBusiness => 6,
            ProductType.FsmTeam => 5,
            ProductType.FsmSolo => 4,
            ProductType.Invoicing => 3,
            ProductType.Premium => 2,
            ProductType.Plus => 1,
            _ => 0
        };
    }
}

### 2. Rename `GetLastSubscription` → `GetPrimarySubscription` and add tier priority

**File**: [`AccountSubscriptionExtensions.cs`](https://github.com/m-unicorn/Invoices.Backend/blob/master/Src/Invoices.Implementation.Services/Subscription/AccountSubscriptionExtensions.cs)

New logic:
1. Filter to active subscriptions (`IsActive == true`)
2. Sort by `ProductType` priority descending (highest tier first)
3. Within the same tier, sort by `ExpirationTime` descending (latest expiration first)
4. Take the first result
5. If no active subscriptions, fall back to the one with the latest `StartTime`

```csharp
public static AccountSubscription? GetPrimarySubscription(
    this IReadOnlyCollection<AccountSubscription> subscriptions)
{
    return subscriptions
               .Where(s => s.IsActive)
               .OrderByDescending(s => ProductTypePriority.GetPriority(s.ProductType))
               .ThenByDescending(s => s.ExpirationTime ?? DateTime.MinValue)
               .FirstOrDefault() ??
           subscriptions.MaxBy(t => t.StartTime);
}
```

## Impact Assessment

This change affects **all API consumers** that call `GET /plans/current` or `GET /api/account/subscription`. The behavior change is:

| Scenario | Before | After |
|----------|--------|-------|
| Active Plus (expires 2027) + Active FsmSolo (expires 2026) | Plus selected | FsmSolo selected |
| Active Premium (expires 2026) + Active Invoicing (expires 2025) | Premium selected | Invoicing selected |
| Single active subscription | That one selected | Same (no change) |
| Multiple same-tier subscriptions | Latest expiration | Same (no change) |

This is the **intended product behavior** — higher-tier subscriptions should always take priority. The risk is low because the bug fix aligns with user expectations (they paid for a higher plan and should get its features).

### Related code that already handles multi-subscription

- `PlanUpgradeService.FindPlanToUpgrade()` — already prioritizes by `ProductKey == "tofu"` for upgrade selection
- `PlansService.GetAllActiveAsync()` — returns all active plans without selection (unaffected)
- `PlansService.GetActiveStripePlanAsync()` — selects Stripe plan for cancellation (may benefit from similar tier priority in future)
