# Step 6.2: Relations

Covers the Estimate ↔ Job ↔ Invoice relation design, deprecation of job-side input linking, job validation on invoice/estimate save, `EstimateId` on Invoice, and invoice removal cleanup. These changes ensure the relation chain stays consistent when any two entities are linked.

## Design Principle

**Ownership lives on invoice and estimate, not on the job.**

- Invoice and estimate hold a `JobId` field that the client sends on create/update requests.
- Job still accepts `InvoiceId`/`EstimateId` on upsert for backward compatibility, but these fields are **deprecated** — new clients should link via invoice/estimate saves instead.
- Job responses still expose `InvoiceId`/`EstimateId` (derived from `Job.Relations`), so clients can read the links.

The preferred linking flow is through invoice/estimate saves, where the existing backend sync mechanisms (`TryUpdateJobSummary`, `TryUpdateJobLink`) propagate the link to `Job.Relations` automatically. The deprecated job-side input path will be removed after all clients are updated (see [6.6](6.6_cleanup_deprecated.md)).

## Current State

| Entity | Field | Direction | Synced to Job |
|--------|-------|-----------|---------------|
| **Job** | `Relations.Estimates[]`, `Relations.Invoices[]`, `Summary.InvoiceId` | Response only | — |
| **Estimate** | `JobId` | Client sends | `Invoices.Api.Services.EstimatesService.TryUpdateJobLink` → `UpdateJobFromEstimateCommand` |
| **Invoice** | `JobId` | Client sends | `Invoices.Api.Services.InvoicesService.TryUpdateJobSummary` → `UpdateJobFromInvoiceCommand` |
| **Estimate** | `InvoiceId` | Cross-link | Set from job's invoice link during save ([§3b](#3b-add-invoiceid-to-estimate)) |
| **Invoice** | `EstimateId` | Client sends | Stored on invoice; used to resolve `JobId` ([§3](#3-add-estimateid-to-invoice)) |

**What already works:**
- Saving an estimate with `JobId` → adds it to `Job.Relations.Estimates`
- Saving an invoice with `JobId` → adds it to `Job.Relations.Invoices` + updates `Summary.InvoiceId`
- Creating a job from estimate ([6.3](6.3_create_job_from_estimate.md)) → links estimate server-side, sets currency, raises events

**Sync scenario:** An invoice or estimate may arrive with a `JobId` before the job itself is synced. This is expected — the save proceeds normally (JobId is stored on the entity), and `Job.Relations` will be populated once the job arrives and subsequent saves trigger `TryUpdateJobSummary` / `TryUpdateJobLink`.

## Invoice ↔ Estimate Link

When a client creates an invoice from an estimate, both `JobId` and `EstimateId` should propagate to the invoice. Conversely, when saving an estimate linked to a job that already has an invoice, `InvoiceId` should propagate to the estimate.

**Resolution:**
- Add `EstimateId` field to the `Invoice` model and DTO (see [§3](#3-add-estimateid-to-invoice)). The client sends `EstimateId` when creating an invoice from an estimate.
- Add `InvoiceId` field to the `Estimate` model and DTO (see [§3b](#3b-add-invoiceid-to-estimate)). Already exists on the Tofu.Invoices proto — needs BFF model + mapping.
- Cross-link resolution happens in `InvoicesService.Add` / `EstimatesService.Add` using the already-fetched job (§2.1 / §2.2).

## Sync Rules

```
Estimate ──── JobId (client sends) ────→ Job.Relations.Estimates    ✅ exists
Estimate ──── InvoiceId (cross-link) ──→ stored on Estimate         🔧 new (§3b)
Invoice  ──── JobId (client sends) ────→ Job.Relations.Invoices     ✅ exists
Invoice  ──── EstimateId (client sends) ────→ stored on Invoice     🔧 new (§3)
Job      ──── InvoiceId/EstimateId ────→ deprecated on input         🔧 change (remove in 6.6)
```

The triangle is maintained by invoice and estimate always sending `JobId` when linked. The job's internal relations are populated automatically by backend sync. When creating an invoice from an estimate, the client sends both `JobId` and `EstimateId` on the invoice.

## Unlink Rules

Job relations are append-only with two exceptions:

| Trigger | What is cleaned up |
|---------|--------------------|
| **Job deletion** | `IJobInvoiceService.ClearInvoiceLinks` (set invoice.JobId = null) + `IJobEstimateService.ClearEstimateLinks` (set estimate.JobId = null) + soft-delete job |
| **Invoice removal** | Remove invoice from `Job.Relations.Invoices`, reset `Job.Summary` (Invoices.Backend). Clear `InvoiceId` on linked estimate (Tofu.Invoices, inside Delete RPC — [6.1 §3](6.1_tofu_invoices_link_job.md)) |

**Note:** In sync scenarios the job may not exist yet when the invoice/estimate is saved. The `JobId` is still persisted on the entity; `Job.Relations` will catch up on subsequent saves once the job is synced. Timeline events are still created with an empty `jobNumber` when the job is not found — the client handles the missing job number in the display text (see [§2](#2-job-validation-and-jobnumber-on-invoiceestimate-save) and [6.1 §2](6.1_tofu_invoices_relations_update.md)).

## 2. Job validation, jobNumber, and cross-link resolution on invoice/estimate save

Before saving an invoice or estimate with a `JobId`, the service fetches the job to:
1. **Validate** the job doesn't already have a different invoice/estimate linked
2. **Extract `jobNumber`** to pass through to the gRPC Add call (Tofu.Invoices uses it for timeline events)
3. **Resolve cross-entity links** from the job's relations (see overview flow diagrams)

### Architecture: Gateway / Service split

The original `InvoicesService` and `EstimatesService` in `Tofu.Invoices` were renamed to **`InvoicesGateway`** and **`EstimatesGateway`** — thin gRPC proxies that only handle request mapping and the gRPC call. They implement `IInvoicesGateway` / `IEstimatesGateway`.

New **`InvoicesService`** and **`EstimatesService`** were created in `Invoices.Api/Services/`. These implement the original `IInvoicesService` / `IEstimatesService` interfaces and add the business logic: job validation, `jobNumber` resolution, cross-link resolution, and post-save job sync (`TryUpdateJobSummary`, `TryUpdateJobLink`). Internally they delegate to the gateway for the actual gRPC call.

**DI registrations:**

| Interface | Implementation | Registered in |
|-----------|---------------|---------------|
| `IInvoicesGateway` | `Tofu.Invoices.InvoicesGateway` | `InvoicesServicesInstaller` |
| `IEstimatesGateway` | `Tofu.Invoices.EstimatesGateway` | `InvoicesServicesInstaller` |
| `IInvoicesService` | `Invoices.Api.Services.InvoicesService` | `CommonServicesConfiguration` |
| `IEstimatesService` | `Invoices.Api.Services.EstimatesService` | `CommonServicesConfiguration` |

Jobs domain services (`JobInvoiceService`, `JobEstimateService`, `JobItemsService`, `GetJobTimelineQueryHandler`) inject the **Gateway** interfaces directly — they manage job links themselves and don't need the API-layer validation.

### Validation logic

The validation logic lives in `InvoicesService.Add` / `EstimatesService.Add` in `Invoices.Api/Services/` (not the controllers), using a private `TryGetJob(accountId, jobId, ct)` helper that returns `JobDto?` (returns null if `jobId` is not a valid GUID or the job is not found).

### 2.1 Invoice save — `InvoicesService.Add`

**File:** `Invoices.Api/Services/InvoicesService.cs`

Before the gRPC Add call:

```csharp
var invoice = request.Invoice;

if (invoice.JobId is not null)
{
    var job = await TryGetJob(invoice.AccountId, invoice.JobId, token);

    if (job != null)
    {
        if (!string.IsNullOrEmpty(job.InvoiceId) && job.InvoiceId != invoice.Id)
            throw new JobAlreadyHasInvoiceException(invoice.JobId, job.InvoiceId);

        request.JobNumber = job.Number;

        // Cross-link: if job has an estimate, set it on the invoice
        if (!string.IsNullOrEmpty(job.EstimateId))
            invoice.EstimateId = job.EstimateId;
    }
}
```

Updating the same invoice (same ID) passes validation. Only a **different** invoice targeting the same job is blocked. If the job is not found, the save proceeds without jobNumber (sync scenario — the job hasn't been synced yet).

### 2.2 Estimate save — `EstimatesService.Add`

**File:** `Invoices.Api/Services/EstimatesService.cs`

Before the gRPC Add call:

```csharp
var estimate = request.Estimate;
var estimateAlreadyLinked = false;

if (estimate.JobId is not null)
{
    var job = await TryGetJob(estimate.AccountId, estimate.JobId, token);

    if (job != null)
    {
        if (!string.IsNullOrEmpty(job.EstimateId) && job.EstimateId != estimate.Id)
            throw new JobAlreadyHasEstimateException(estimate.JobId, job.EstimateId);

        request.JobNumber = job.Number;
        estimateAlreadyLinked = job.EstimateId == estimate.Id;

        // Cross-link: if job has an invoice, set it on the estimate
        if (!string.IsNullOrEmpty(job.InvoiceId))
            estimate.InvoiceId = job.InvoiceId;
    }
}
```

After the gRPC Add call, **skip `TryUpdateJobLink` if the estimate is already linked** — `UpdateJobFromEstimateCommand` would fetch the job again and `TryAddEstimateLink` would be a no-op:

```csharp
if (!estimateAlreadyLinked)
    await TryUpdateJobLink(result.Estimate, token);
```

**Note:** The invoice side (`TryUpdateJobSummary`) cannot be skipped even when the link exists, because `UpdateJobFromInvoiceCommand` always updates `Job.Summary` (amounts, status) which may change on every invoice save.

### 2.3 TryGetJob helper

**File:** `Invoices.Api/Services/InvoicesService.cs` and `Invoices.Api/Services/EstimatesService.cs`

Both services use an identical private helper:

```csharp
/// <summary> Returns the job linked to the given jobId, or null if not found or jobId is not a valid GUID. </summary>
private async Task<JobDto?> TryGetJob(string accountId, string jobId, CancellationToken ct)
{
    if (!Guid.TryParse(jobId, out var jobGuid))
        return null;

    try
    {
        var result = await _handlerDispatcher.DispatchQuery<GetJobByIdQuery, GetJobByIdResult>(
            new GetJobByIdQuery(accountId, jobGuid), ct);
        return result.Job;
    }
    catch (EntityNotFoundException)
    {
        // Sync scenario — job not yet synced
        return null;
    }
}
```

## 4. Remove Invoice — job cleanup

When an invoice is deleted, the backend cleans up the job's relation and summary so the job behaves as if it never had an invoice.

### 4.1 `InvoicesService.Delete`

**File:** `Invoices.Api/Services/InvoicesService.cs`

After deleting the invoice via the gateway gRPC call (which handles clearing `InvoiceId` on the linked estimate — see [6.1 §3](6.1_tofu_invoices_link_job.md)), `TryClearJobLinksForDeletedInvoice` is called unconditionally with `(accountId, invoiceId)`. The `ClearJobInvoiceLinkCommand` handler finds the job (if any) that has this invoice linked and clears the relation + resets the summary. If no job references the invoice, the command is a no-op.

```csharp
public async Task<DeleteInvoiceResponseModel> Delete(DeleteInvoiceRequestModel request, string traceId,
    CancellationToken token)
{
    var result = await _gateway.Delete(request, traceId, token);
    await TryClearJobLinksForDeletedInvoice(request.AccountId, request.InvoiceId, token);
    return result;
}
```

### 4.1b `EstimatesService.Delete`

**File:** `Invoices.Api/Services/EstimatesService.cs`

For estimate deletion, the service pre-fetches the estimate via `_gateway.Find` to get the `JobId`, then deletes via the gateway, then clears the job link only if a valid `JobId` was present:

```csharp
var existing = await _gateway.Find(new GetEstimateRequestModel { ... }, traceId, token);
var result = await _gateway.Delete(request, traceId, token);

var jobId = existing?.Estimate.JobId;
if (!string.IsNullOrEmpty(jobId) && Guid.TryParse(jobId, out var jobGuid))
    await TryClearJobLinksForDeletedEstimate(request.AccountId, jobGuid, request.EstimateId, token);
```

### 4.2 Estimate side

The estimate's `InvoiceId` is cleared by Tofu.Invoices during the invoice Delete RPC ([6.1 §3](6.1_tofu_invoices_link_job.md)). No additional action needed from Invoices.Backend.
