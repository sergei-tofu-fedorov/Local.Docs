# Step 6.6: Cleanup Deprecated Job Input Linking

**Prerequisite:** All clients (web, mobile) have been updated to use the new invoice/estimate save flows for linking. No client sends `InvoiceId`/`EstimateId` on job upsert requests.

This step removes the deprecated backward-compatibility code that was kept in [6.2](6.2_relations.md) during the transition period.

## 1. `Invoices.Api/Dto/Jobs/JobUpsertRequestDto.cs`

Remove `InvoiceId` and `EstimateId` properties:

```csharp
// REMOVE:
[Obsolete("Use invoice/estimate save with JobId instead")]
public string? InvoiceId { get; set; }

[Obsolete("Use invoice/estimate save with JobId instead")]
public string? EstimateId { get; set; }
```

## 2. `Invoices.Api/Dto/Jobs/JobsApiMappings.cs` — `ToCommand()`

Remove the two field mappings:

```csharp
// REMOVE:
InvoiceId: api.InvoiceId,
EstimateId: api.EstimateId,
```

## 3. `Jobs.Contracts/Jobs/Commands/UpsertJobCommand.cs`

Remove the two parameters from the command record:

```csharp
// REMOVE:
string? InvoiceId,
string? EstimateId,
```

## 4. `Jobs.Application/Commands/UpsertJobCommandHandler.cs`

Remove `FetchAndApplyRelations` method and its call:

```csharp
// REMOVE call in ApplyUpdates():
await FetchAndApplyRelations(job, command, occurredAt, ct);

// REMOVE entire method:
private async Task FetchAndApplyRelations(Job job, UpsertJobCommand command, ...) { ... }
```

After removing `FetchAndApplyRelations`, the handler no longer uses `IJobInvoiceService` or `IJobEstimateService`. Remove them from the constructor:

```csharp
// REMOVE these two fields + constructor parameters:
private readonly IJobInvoiceService _jobInvoiceService;
private readonly IJobEstimateService _jobEstimateService;
```

**Note:** `IJobInvoiceService` and `IJobEstimateService` interfaces stay in the codebase — they are still used by `DeleteJobCommandHandler`, `CreateJobFromEstimateCommandHandler`, and invoice removal cleanup.
