# 4.7 Jobs Without Client

## Overview

The mobile app sometimes sends an upsert-job request before the client has been synced to the backend. Previously, creating a new job called `IJobClientsService.GetClient()` which throws `ClientNotFoundException` if the client doesn't exist yet. This blocked job creation entirely.

**Goal**: Allow job creation when the referenced client hasn't been synced yet. Audit all code paths that assume the client exists for a job.

---

## Current State (before fix)

| Operation | Client check | Behaviour when client missing |
|-----------|-------------|-------------------------------|
| **Create job** (upsert, new) | `GetClient()` ‚Äî throws | **404 ClientNotFoundException** |
| **Update job** (upsert, existing) | None | Works fine |
| **Get job details** (`GET /jobs/{id}`) | `GetRelatedData()` ‚Üí `GetClient()` ‚Äî throws | **404 ClientNotFoundException** (by design when no snapshot) |
| **Sync jobs** (`GET /jobs/sync`) | `FindClients()` ‚Üí `GetClientsByIds()` ‚Äî silent skip | Works fine (missing clients omitted) |
| **Store snapshot** (job completion) | `GetClient()` ‚Äî throws | **404 ClientNotFoundException** |

### Root Cause: Create Path

`UpsertJobCommandHandler.ApplyUpdates` (line 73):
```csharp
var client = await _jobClientsService.GetClient(command.AccountId, command.ClientId, ct);
var title = command.Title ?? client.Name;   // uses client name as default title
```

The `GetClient` call existed for two reasons:
1. Validate that the client exists
2. Use `client.Name` as fallback title when `command.Title` is null

---

## Affected Code Paths

### 1. Job Creation ‚Äî `UpsertJobCommandHandler` (fixed)

**File:** `Src/Jobs/Jobs.Application/Commands/UpsertJobCommandHandler.cs:71-78`

**Problem:** `GetClient()` throws when client doesn't exist yet.

**Fix:** Replaced `GetClient` with `TryGetClient`. If the client is not found, fall back to `command.Title`. If both are null ‚Äî throw `InvalidOperationException`.

```csharp
var client = await _jobClientsService.TryGetClient(command.AccountId, command.ClientId, ct);
var jobNumber = command.Number ?? await _jobsRepository.NextNumber(command.AccountId, ct);
var title = command.Title ?? client?.Name
    ?? throw new InvalidOperationException("Title is required when the client has not been synced yet.");

job = new Job(jobId, command.AccountId, jobNumber, title, command.ClientId, occurredAt);
```

Title resolution order: `command.Title` ‚Üí `client.Name` ‚Üí throw.

### 2. Client Snapshot on Completion ‚Äî `UpsertJobCommandHandler` (fixed)

**File:** `Src/Jobs/Jobs.Application/Commands/UpsertJobCommandHandler.cs:98-110`

**Problem:** When a job transitions to completed status, `GetClient()` was called to freeze a snapshot. If the client still hadn't been synced, this threw.

**Fix:** Use `TryGetClient` instead. If the client exists, store the snapshot; if not, skip it.

```csharp
if (snapshotRequired)
{
    var client = await _jobClientsService.TryGetClient(command.AccountId, command.ClientId, ct);
    if (client != null)
        job.StoreClientSnapshot(client.ToSnapshot());
}
```

If the client hasn't been synced yet at completion time, the snapshot is omitted. It can be backfilled later if needed.

### 3. Get Job Details ‚Äî `JobDetailsService.GetRelatedData` (no change)

**File:** `Src/Invoices.Api/Services/JobDetailsService.cs:31-36`

**Behaviour:** When `job.ClientSnapshot == null`, `GetClient()` is called to fetch the client. If the client doesn't exist, `ClientNotFoundException` propagates and the request fails. This is **by design** ‚Äî if a job has no `ClientSnapshot` and the client cannot be retrieved, the request should fail. The caller should not receive partial data with a silently missing client.

```csharp
ClientDto? client = null;
if (job.ClientSnapshot == null)
{
    var clientEntity = await _clientsService.GetClient(accountId, job.ClientId, includeCalculations: false, ct);
    client = MapToClientDto(clientEntity);
}
```

**Note:** This uses `IClientsService.GetClient` (API layer), not `IJobClientsService.TryGetClient` (Jobs domain). The distinction is intentional ‚Äî the API layer enforces the client exists when returning full job details.

### 4. Sync Jobs ‚Äî `JobDetailsService.FindClients` (already safe)

**File:** `Src/Invoices.Api/Services/JobDetailsService.cs:55-68`

Uses `GetClientsByIds()` which silently omits missing clients from the dictionary result. Jobs with `ClientSnapshot != null` are excluded from the client lookup entirely. **No change needed.**

### 5. `IJobClientsService.TryGetClient` (added)

**File:** `Src/Jobs/Jobs.Application/Services/Clients/IJobClientsService.cs`

Added a nullable variant that returns `null` instead of throwing:

```csharp
/// <summary> Returns null instead of throwing when the client is not found. </summary>
Task<JobClient?> TryGetClient(string accountId, string clientId, CancellationToken ct);
```

**Implementation** (`JobClientsService.cs`): wraps `GetClient` in a try/catch for `ClientNotFoundException`:

```csharp
public async Task<JobClient?> TryGetClient(string accountId, string clientId, CancellationToken ct)
{
    try
    {
        return await GetClient(accountId, clientId, ct);
    }
    catch (ClientNotFoundException)
    {
        return null;
    }
}
```

## Design Notes

- **No schema changes** ‚Äî `ClientId` remains required on `JobUpsertRequestDto`. The job always has a `clientId`, but the client entity may not exist yet.
- **Title is required when client is missing** ‚Äî When client is not found during creation, `command.Title` is used. If both are null, `InvalidOperationException` is thrown ‚Äî a job cannot be created without a name.
- **Snapshot on completion** ‚Äî The handler uses `TryGetClient` to fetch the client. If the client exists, the snapshot is stored; if not, it is skipped. No changes to the upsert request contract.
- **Get job details fails if client is missing** ‚Äî When a job has no `ClientSnapshot` and the client cannot be retrieved, `GET /jobs/{id}` returns `ClientNotFoundException`. This is intentional: the caller should not silently receive incomplete data.
- **Backward compatible** ‚Äî No API contract changes. The `client` field in job details can already be `null`.
