# 4.6 Include Client Info in Job Sync

## Overview

Add a flag to the job sync endpoint to optionally include live client information in the response. This enables mobile apps to sync jobs with client data in a single request.

## Current State

| Aspect | Current Implementation |
|--------|------------------------|
| Endpoint | `GET /jobs/sync?cursor={cursor}&limit={limit}` (v3.0) |
| Response | `SyncResponseDto<JobDto>` with Items, NextCursor, HasMore |
| Client data | Only `ClientId` and `ClientSnapshot` (frozen at completion) in JobDto |
| Live client | Must call separate `GET /clients/{id}` endpoint |

### Current Response Structure

```json
{
  "items": [
    {
      "itemId": "job-guid",
      "change": "updated",
      "item": {
        "id": "job-guid",
        "clientId": "client-id",
        "clientSnapshot": null,
        // ... other job fields
      }
    }
  ],
  "nextCursor": "...",
  "hasMore": true,
  "delayNextRequestInSeconds": 0
}
```

---

## Solution

Add `includeRelations` query parameter. When set, the response includes a `relations` object containing a `clients` array. Each client is matched to jobs via `clientId`. `JobDto` stays unchanged. Enrichment is handled entirely in the controller via `_jobDetailsService` â€” the query handler is not modified.

### Target Response Structure

```json
{
  "items": [
    {
      "itemId": "job-guid",
      "change": "updated",
      "item": {
        "id": "job-guid",
        "clientId": "client-id",
        "clientSnapshot": null,
        // ... other job fields (unchanged)
      }
    }
  ],
  "relations": {                            // NEW - only when includeRelations=true
    "clients": [
      {
        "clientId": "client-id",
        "name": "John Doe",
        "phone": "+1234567890",
        "email": "john@example.com",
        "address": "123 Main St"
      }
    ]
  },
  "nextCursor": "...",
  "hasMore": true,
  "delayNextRequestInSeconds": 0
}
```

---

## Implementation

### 1. Create Generic SyncResponseDto with Relations

**File:** `Src/Invoices.Api/Dto/SyncResponseDto.cs`

Add a second generic overload that carries a typed `Relations` object:

```csharp
// Existing (unchanged)
public class SyncResponseDto<T> where T : class
{
    public required IReadOnlyList<SyncChangeItemDto<T>> Items { get; init; }
    public required string? NextCursor { get; init; }
    public required bool HasMore { get; init; }
    public required int DelayNextRequestInSeconds { get; init; }
}

// NEW
public class SyncResponseDto<T, TRelations> : SyncResponseDto<T>
    where T : class
    where TRelations : class
{
    /// <summary> Related data for items in this page. Null when relations are not requested. </summary>
    [JsonProperty(NullValueHandling = NullValueHandling.Ignore)]
    public TRelations? Relations { get; init; }
}
```

**File:** `Src/Invoices.Api/Dto/Jobs/JobSyncRelationsDto.cs`

Define the relations shape for the jobs sync endpoint:

```csharp
public class JobSyncRelationsDto
{
    public required IReadOnlyList<ClientDto> Clients { get; init; }
}
```

`JobDto` stays unchanged â€” no new properties, no inheritance. No DTO inheritance used.

### 2. Add Batch Client Lookup to JobDetailsService

**File:** `Src/Invoices.Api/Services/IJobDetailsService.cs`

Add a new method for batch client lookups:

```csharp
public interface IJobDetailsService
{
    Task<(ClientDto? Client, InvoiceDto? Invoice, EstimateDto? Estimate)> GetRelatedData(
        ContractJobs.JobDto job, string accountId, CancellationToken ct);

    /// <summary> Fetch ClientDto for each client ID. Missing clients are omitted. </summary>
    Task<Dictionary<string, ClientDto>> GetClientsByIds(
        string accountId, IEnumerable<string> clientIds, CancellationToken ct);
}
```

**File:** `Src/Invoices.Api/Services/JobDetailsService.cs`

Use the existing `_clientsService.GetClientsByIds` batch method and reuse the `ManageableClient â†’ ClientDto` mapping logic from `GetClientDto`:

```csharp
public async Task<Dictionary<string, ClientDto>> GetClientsByIds(
    string accountId, IEnumerable<string> clientIds, CancellationToken ct)
{
    var clients = await _clientsService.GetClientsByIds(accountId, clientIds, ct);

    return clients.ToDictionary(
        kvp => kvp.Key,
        kvp => MapToClientDto(kvp.Value));
}

/// <summary> Shared mapping: extract "Main" info and build ClientDto. </summary>
private static ClientDto MapToClientDto(ManageableClient client)
{
    var mainInfo = client.Info.FirstOrDefault(i => i.Type == ClientInfoType.Main)
                   ?? client.Info.FirstOrDefault();

    return new ClientDto(
        mainInfo?.Name, mainInfo?.Phone, mainInfo?.Email, mainInfo?.Address, client.ClientId);
}
```

Then refactor `GetClientDto` (used by `GetRelatedData`) to reuse the same mapping. Returns `null` instead of throwing when `mainInfo` is missing â€” this allows batch callers to silently skip clients without valid info:

```csharp
private async Task<ClientDto?> GetClientDto(string accountId, string clientId, CancellationToken ct)
{
    var client = await _clientsService.GetClient(accountId, clientId, includeCalculations: false, ct);

    var mainInfo = client.Info.FirstOrDefault(i => i.Type == ClientInfoType.Main)
                   ?? client.Info.FirstOrDefault();
    if (mainInfo == null)
        return null;

    return MapToClientDto(client);
}
```

### 3. Update Controller

**File:** `Src/Invoices.Api/Controllers/JobsController.cs`

Change the return type to `SyncResponseDto<JobDto, JobSyncRelationsDto>` and add `includeRelations` query parameter. The query handler is unchanged â€” enrichment happens in the controller via `_jobDetailsService`:

```csharp
[HttpGet("sync")]
[MapToApiVersion("3.0")]
public async Task<ActionResult<SyncResponseDto<JobDto, JobSyncRelationsDto>>> Sync(
    [FromQuery] string? cursor = null,
    [FromQuery] int limit = 100,
    [FromQuery] bool includeRelations = false,  // NEW
    CancellationToken ct = default)
{
    var query = new SyncJobsQuery(AccountId: AccountId, PageSize: limit, Cursor: cursor);

    var result = await _dispatcher.DispatchQuery<SyncJobsQuery, SyncJobsResult>(query, ct);

    var items = result.Items.Select(i => new SyncChangeItemDto<JobDto>
    {
        ItemId = i.ItemId,
        Change = i.Change.ToDto(),
        Item = i.Item?.ToApi()
    }).ToList();

    // NEW: Build relations if requested
    JobSyncRelationsDto? relations = null;
    if (includeRelations)
    {
        // Skip jobs that already have a ClientSnapshot (completed jobs with frozen client info)
        var clientIds = items
            .Where(i => i.Item is { ClientSnapshot: null } && !string.IsNullOrEmpty(i.Item.ClientId))
            .Select(i => i.Item!.ClientId)
            .Distinct();

        var clientsDict = await _jobDetailsService.GetClientsByIds(AccountId, clientIds, ct);

        relations = new JobSyncRelationsDto
        {
            Clients = clientsDict.Values.ToList()
        };
    }

    return new SyncResponseDto<JobDto, JobSyncRelationsDto>
    {
        Items = items,
        Relations = relations,
        NextCursor = result.NextCursor,
        HasMore = result.HasMore,
        DelayNextRequestInSeconds = _options.DelayNextRequestInSeconds
    };
}
```

## Design Notes

- **Generic relations pattern:** `SyncResponseDto<T, TRelations>` extends the existing `SyncResponseDto<T>` with a typed `Relations` property. No DTO inheritance needed â€” each sync endpoint defines its own relations class (e.g. `JobSyncRelationsDto`). Reusable for other sync endpoints that need related data.
- **No DTO changes to JobDto:** Client data lives in the `relations` object, not inside each job. Jobs reference clients via `clientId` as before.
- **No query handler changes:** `SyncJobsQuery` and `SyncJobsQueryHandler` are untouched. Enrichment is a controller-level concern handled by `_jobDetailsService`.
- **Batch fetch via existing service:** `GetClientsByIds` calls `_clientsService.GetClientsByIds` which already exists on `IClientsService`. No new repository methods required.
- **Shared mapping:** `ManageableClient â†’ ClientDto` logic is extracted into a `MapToClientDto` helper, reused by both `GetClientsByIds` and `GetClientDto` (called by `GetRelatedData`).
- **Null-tolerant client lookup:** `GetClientDto` returns `null` instead of throwing when `mainInfo` is missing. This prevents a single client with no info from failing the entire batch/sync request â€” the client is simply omitted from results.
- **No new controller dependencies:** `JobsController` already injects `IJobDetailsService`, so no constructor changes are needed.
- **Backward compatible JSON:** When `includeRelations=false`, the `relations` field is omitted from the response (`NullValueHandling.Ignore`). Existing callers see no change.
- **No duplication:** Each unique client appears once in `relations.clients`, regardless of how many jobs reference it.

## API Changes Summary

| Endpoint | Change |
|----------|--------|
| `GET /jobs/sync` | New query param: `?includeRelations=true` |

### Request Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `cursor` | string | null | Pagination cursor |
| `limit` | int | 100 | Page size |
| `includeRelations` | bool | false | **NEW** - Include related data (clients) |

### Response Changes

Return type changes from `SyncResponseDto<JobDto>` to `SyncResponseDto<JobDto, JobSyncRelationsDto>`. When `includeRelations=false` the JSON shape is identical (the `relations` field is omitted). When `includeRelations=true`:

| Field | Type | Description |
|-------|------|-------------|
| `relations` | `JobSyncRelationsDto?` | Related data object (omitted when null) |
| `relations.clients` | `ClientDto[]` | Unique clients referenced by jobs in this page. Match to jobs via `clientId`. |

---

## Performance Considerations

1. **Batch fetch:** All unique client IDs are fetched in a single `GetClientsByIds` call
2. **Distinct IDs:** Only unique client IDs are queried (deduped before fetching)
3. **No duplication:** Each client appears once regardless of how many jobs reference it
4. **Deleted jobs:** Skipped (no client lookup needed)
5. **Missing clients:** Clients not found are omitted from the array
