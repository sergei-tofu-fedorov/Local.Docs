# 4.8 Enrich Invoice/Estimate Events with DocNumber

## Overview

Job timeline activity phrasing references the document number (e.g. "You created Invoice #1001 for this job", "You sent Estimate #E-1001 to email"). Currently, invoice and estimate event payloads do not include the document number. The frontend must make separate calls or guess.

**Goal**: Add a nullable `docNumber` field to all invoice and estimate event payloads. The value comes from the `Number` property on the `Invoice` / `Estimate` entity at the time the event is created.

**Source of truth**: `features/timeline/Timeline Events.tsv`

---

## Previous State

| Aspect | Before |
|--------|--------|
| Domain events | No document number — concrete events inherited directly from `DomainEvent` / `DomainEvent<T>` |
| Invoice event payloads | `InvoiceStatusChangedPayload`, `InvoiceEmailStatusChangedPayload`, `InvoicePaymentReceivedPayload` — no document number |
| Estimate event payloads | `StatusChangedPayload`, `EmailStatusChangedPayload` — no document number |
| Event factory | `InvoiceEventsFactory` / `EstimateEventsFactory` — had access to `entity.Number` but did not include it in payloads |
| Activity phrasing | References `Invoice #[N]` and `Estimate #[N]` — frontend had no source for `[N]` from event payload |

### Affected Events (per TSV)

All invoice and estimate events carry `docNumber` in their payload:

| Event Type | Payload Record |
|------------|----------------|
| `invoiceStatusChanged` | `InvoiceStatusChangedPayload` |
| `invoiceEmailStatusChanged` | `InvoiceEmailStatusChangedPayload` |
| `invoicePaymentReceived` | `InvoicePaymentReceivedPayload` |
| `estimateStatusChanged` | `StatusChangedPayload` |
| `estimateEmailStatusChanged` | `EmailStatusChangedPayload` |

### Current Payload Examples

**InvoiceStatusChanged:**
```json
{ "from": null, "to": "notPaid", "provider": null }
```

**InvoiceEmailStatusChanged:**
```json
{ "from": null, "to": "sent", "recipient": "john@example.com" }
```

**EstimateStatusChanged:**
```json
{ "from": "draft", "to": "sent", "sentMethod": "email" }
```

---

## Solution

Two layers of change:

1. **Domain events carry `DocNumber`.** Intermediate abstract classes (`InvoiceDomainEvent` / `EstimateDomainEvent`, generic and non-generic) sit between the base `DomainEvent` / `DomainEvent<T>` and the concrete event classes. Each intermediate class adds a `string? DocNumber` property. `EnrichedInvoice` and `EnrichedEstimate` set `DocNumber` at every event-creation site.

2. **Payload records inherit from a base with `DocNumber`.** `InvoiceEventPayload` and `EstimateEventPayload` each define `DocNumber`. All concrete payload records inherit from them. The factory `Build` methods read `domainEvent.DocNumber` and set it directly on the payload — no separate enrichment step needed.

### Target Payload Examples

**InvoiceStatusChanged:**

**InvoiceEmailStatusChanged:**

## Implementation

### 1. Intermediate Domain Event Base Classes

**File:** `Tofu.Invoices.Domain/Models/Events/DomainEvent.cs`

Four new abstract classes inserted between the base `DomainEvent` / `DomainEvent<T>` and the concrete event classes. Each adds `string? DocNumber`.

```
DomainEvent
├── DomainEvent<T>
│   ├── InvoiceDomainEvent<T>        (has DocNumber)
│   │   ├── InvoiceStatusChangeDomainEvent
│   │   ├── InvoiceMailStatusChangeDomainEvent
│   │   └── InvoicePaymentReceivedDomainEvent
│   └── EstimateDomainEvent<T>       (has DocNumber)
│       ├── EstimateStatusChangeDomainEvent
│       └── EstimateMailStatusChangeDomainEvent
├── InvoiceDomainEvent               (has DocNumber)
│   └── InvoiceCreatedDomainEvent
└── EstimateDomainEvent              (has DocNumber)
    └── EstimateCreatedDomainEvent
```

### 2. Set DocNumber at Event-Creation Sites

**Files:** `EnrichedInvoice.cs`, `EnrichedEstimate.cs`

Every `RegisterDomainEvent(new ...DomainEvent { ... })` call now includes `DocNumber = Entity.Number` (or `invoice.Number` / `estimate.Number` in `Create()`).

**Critical ordering fix in `EnrichedInvoice.Update()`:** `Entity.Number = newInvoice.Number` is now set **before** `CheckPaymentReceived()` so that payment events capture the current document number.

### 3. Invoice and Estimate Base Payloads

**Files:** `InvoiceEvent.cs`, `EstimateEvent.cs`

Base payload records (`InvoiceEventPayload`, `EstimateEventPayload`) with `DocNumber` — unchanged from previous implementation. All concrete payload records inherit from them.

### 4. Simplified Event Factories

**Files:** `InvoiceEventsFactory.cs`, `EstimateEventsFactory.cs`

Factories no longer capture `entity.Number` or thread a `docNumber` parameter. Each `Build` method reads `domainEvent.DocNumber` and sets it directly on the payload record. `SerializePayload` simply serializes — no enrichment needed.

```csharp
// Before (factory enrichment):
var docNumber = entity.Number;
BuildInvoiceEvent(domainEvent, ..., docNumber);
// SerializePayload(payload, docNumber) → payload with { DocNumber = docNumber }

// After (domain event carries DocNumber):
BuildInvoiceEvent(domainEvent, ...);
// payload.DocNumber = domainEvent.DocNumber → SerializePayload(payload)
```

---

## Design Notes

- **Intermediate base classes.** `InvoiceDomainEvent` / `InvoiceDomainEvent<T>` and `EstimateDomainEvent` / `EstimateDomainEvent<T>` sit between the framework `DomainEvent` types and the concrete event classes. Each adds `string? DocNumber`. Future per-entity-type fields can be added to these bases.
- **DocNumber set at creation site.** `EnrichedInvoice` and `EnrichedEstimate` set `DocNumber` when creating each domain event. The value is captured at the moment the event occurs, not deferred to the factory.
- **Base payload records.** `InvoiceEventPayload` and `EstimateEventPayload` also define `DocNumber`. Factory `Build` methods copy `domainEvent.DocNumber` onto the payload before serialization.
- **Simplified factories.** Factories no longer thread a `docNumber` parameter through `Build` methods. `SerializePayload` is a plain serializer — no `with` enrichment step.
- **Nullable.** `DocNumber` is nullable because `Number` on Invoice/Estimate is nullable (e.g. draft invoices before a number is assigned).
- **Additive JSON change.** Adding a new field to the serialized payload is backward-compatible — consumers that don't know about `docNumber` will ignore it.
- **No factory signature changes.** The `CreateInvoiceEvents` / `CreateEstimateEvents` public signatures stay the same.
- **Hash stability.** The `docNumber` is part of the serialized payload, so it becomes part of the event hash. This is expected — it's a new field for new events, not a retroactive change.
- **Polymorphic serialization.** `JsonSerializer.Serialize(payload, payload.GetType(), ...)` ensures the derived record's properties are included. Without the explicit `GetType()`, `System.Text.Json` would only serialize base-type properties.

---

## Migration

Existing events will **not** have `docNumber` in their payloads. This is acceptable:
- New events will include `docNumber` when the entity has a `Number` value.
- Frontend should handle missing `docNumber` gracefully (fall back to not showing a number, or fetching it separately).
- Optional: a backfill script could enrich historical events by looking up `Number` from the invoice/estimate entity by `EntityId`.

---

## API Changes

No endpoint changes. The `payload` field in `TimelineItemDto` already contains a JSON string. The new `docNumber` field appears inside that JSON for newly created events.
