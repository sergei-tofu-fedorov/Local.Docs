# 4.2 Enrich Job Events with Worker Name

## Overview

Job events (visit created, visit status changed) need worker names for activity text display. After 4.1, domain events already carry `WorkerName` (set to `null`) and `VisitCount` fields. This stage resolves worker names at the application layer before saving events.

## Current State (after 4.1)

| Aspect | Implementation |
|--------|---------------|
| `VisitCreatedPayload` | Has `WorkerId`, `WorkerName` (null), `VisitCount` |
| `VisitStatusChangedPayload` | Has `WorkerName` (null), no `WorkerId` |
| Domain event factories | Accept `workerName` parameter, pass through to payload |
| Worker resolution | Not yet implemented - all events have `workerName: null` |

### Current Payload Structures (from 4.1)

**VisitCreated:**
```json
{ "visitId": "guid", "scheduledDate": "2024-01-15T09:00:00Z", "workerId": null, "workerName": null, "visitCount": 1 }
```

**VisitStatusChanged:**
```json
{ "visitId": "guid", "previousStatus": "scheduled", "newStatus": "in_progress", "workerName": null }
```

---

## Solution

Resolve worker names at the application layer and pass them into domain event factory methods. The domain layer already accepts `workerName` parameters (from 4.1) - this stage provides actual values instead of `null`.

### When to Populate `workerName`

| Event | Condition | workerName | Activity Text |
|-------|-----------|------------|---------------|
| `visitCreated` | With worker assigned | Worker's name | "You scheduled/created a visit for {Worker name}..." |
| `visitCreated` | Without worker | `null` | "You scheduled/created a visit on..." |
| `visitStatusChanged` | **Worker** changed status | Worker's name | "{Worker name} started/completed the visit" |
| `visitStatusChanged` | **Manager** changed status | `null` | "You started/completed the visit" |

**Key rule:** For `visitStatusChanged`, populate `workerName` **only when the worker app initiated the change**, not when a manager changes status through the main app.

---

## Implementation

**Approach:** Domain event factories already accept `workerName` (from 4.1) but all callers currently pass `null`. This stage:
1. Adds a `workerName` parameter to `Job.UpdateVisits` so visit-created events can carry the name
2. Creates a worker resolution service at the application layer
3. Updates command handlers to resolve names and pass them into domain methods

### 1. Event Payloads & Factories - No Changes

### 3. Create Worker Resolution Service

**File:** `Src/Jobs/Jobs.Application/Services/Workers/IJobWorkerService.cs`

```csharp
public interface IJobWorkerService
{
    /// <summary> Get worker name by ID. Returns null if worker not found. </summary>
    Task<string?> GetWorkerName(string accountId, string? workerId, CancellationToken ct);

    /// <summary> Get worker names for multiple IDs in a single batch. </summary>
    Task<Dictionary<string, string>> GetWorkerNames(
        string accountId,
        IEnumerable<string> workerIds,
        CancellationToken ct);
}
```

**File:** `Src/Jobs/Jobs.Application/Services/Workers/JobWorkerService.cs`

```csharp
public class JobWorkerService : IJobWorkerService
{
    private readonly ITofuAuthApiClient _authClient;
    private readonly ILogger<JobWorkerService> _logger;

    public JobWorkerService(
        ITofuAuthApiClient authClient,
        ILogger<JobWorkerService> logger)
    {
        _authClient = authClient;
        _logger = logger;
    }

    public async Task<string?> GetWorkerName(
        string accountId,
        string? workerId,
        CancellationToken ct)
    {
        if (string.IsNullOrEmpty(workerId) || !Guid.TryParse(workerId, out var workerGuid))
            return null;

        try
        {
            var user = await _authClient.GetTenantUserAsync(accountId, workerGuid, ct);
            return user?.DisplayName ?? user?.Email;
        }
        catch (Exception ex)
        {
            _logger.LogWarning(ex, "Failed to get worker name for {WorkerId}", workerId);
            return null;
        }
    }

    public async Task<Dictionary<string, string>> GetWorkerNames(
        string accountId,
        IEnumerable<string> workerIds,
        CancellationToken ct)
    {
        var result = new Dictionary<string, string>();
        var validIds = workerIds
            .Where(id => !string.IsNullOrEmpty(id))
            .Distinct()
            .ToList();

        if (validIds.Count == 0) return result;

        try
        {
            var users = await _authClient.GetTenantUsersAsync(accountId, ct);
            foreach (var user in users)
            {
                var userId = user.Id.ToString();
                if (validIds.Contains(userId))
                {
                    result[userId] = user.DisplayName ?? user.Email ?? userId;
                }
            }
        }
        catch (Exception ex)
        {
            _logger.LogWarning(ex, "Failed to batch fetch worker names");
        }

        return result;
    }
}
```

```csharp
public class WorkerUpdateVisitStatusCommandHandler
    : ICommandHandler<WorkerUpdateVisitStatusCommand, WorkerUpdateVisitStatusResult>
{
    private readonly IJobsRepository _jobsRepository;
    private readonly IJobDomainEventService _jobDomainEventService;
    private readonly IJobWorkerService _workerService;  // NEW
    private readonly IUnitOfWork _unitOfWork;

    public async Task<WorkerUpdateVisitStatusResult> Handle(
        WorkerUpdateVisitStatusCommand cmd, CancellationToken ct)
    {
        var job = await _jobsRepository.GetJobForUpdate(
                cmd.AccountId,
                JobSpecifications.ByVisitId(cmd.VisitId),
                ct)
            ?? throw new EntityNotFoundException($"Visit '{cmd.VisitId}' not found");

        job.ValidateVersion(cmd.Version);

        var now = DateTimeOffset.UtcNow;
        var workerName = await _workerService.GetWorkerName(cmd.AccountId, cmd.WorkerId, ct);
        var result = job.TryUpdateVisitStatusByWorker(cmd.VisitId, cmd.Status.ToDomain(), workerName, now);

        if (result.Status == VisitUpdateStatus.Blocked)
            return WorkerUpdateVisitStatusResult.Blocked(result.BlockingVisitId!.Value);

        if (result.Status == VisitUpdateStatus.Success)
        {
            _jobsRepository.Update(job, now);
            _jobDomainEventService.Save(job, cmd.WorkerId, ActorType.User, now);
            await _unitOfWork.SaveChangesAsync(ct);
        }

        return WorkerUpdateVisitStatusResult.Ok();
    }
}
```

## API Changes

No API contract changes required. The `Payload` field already contains JSON that clients parse. The `workerName` field was added in 4.1 (with null values) and is already additive.

### Timeline Response Example

```json
{
  "items": [
    {
      "id": 123,
      "entityId": "job-guid",
      "eventType": "visitStatusChanged",
      "payload": "{\"visitId\":\"visit-guid\",\"previousStatus\":\"scheduled\",\"newStatus\":\"completed\",\"workerName\":\"John Smith\"}"
    }
  ]
}
```

---

## Migration

Existing events will have `workerName: null` in their payloads. This is acceptable:
- New events will include worker name when applicable
- Clients should handle missing `workerName` gracefully
- Optional: Backfill script can enrich historical events
