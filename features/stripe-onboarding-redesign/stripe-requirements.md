# Что пользователь должен заполнить и когда это проверяется

Практический справочник: какие поля нужны, чтобы включились платежи и выплаты, как Stripe их проверяет и в какой момент. Для **US, `business_type=individual`, Standard** — наша конфигурация. Родительский док: [README.md](README.md); откуда данные: [prod-funnel.md](prod-funnel.md); откуда что префиллить: [prefill-fields.md](prefill-fields.md).

Источники: [required-verification-information](https://docs.stripe.com/connect/required-verification-information) + замеры на 21 489 live-аккаунтах и на тестовой платформе (2026-07-15).

> **Разметка источников.** `[дока]` — цитата из документации Stripe · `[тест]` — проверено на тестовой платформе спайком · `[прод]` — замер на 21 489 live-аккаунтах · `[код]` — прочитано в исходниках · `[инференс]` — вывод, а не измерение.

## Главное: требований два класса, и путают именно это

| Класс | Что это | Где перечислен |
|---|---|---|
| **Identity (KYC)** | кто вы: имя, DOB, SSN, домашний адрес | официальный список Stripe |
| **Business (ToS/андеррайтинг)** | что вы продаёте: сайт, описание, индустрия, support-телефон | **в списке identity его нет** — приходит из ToS |

Оба гасят `card_payments`. Ошибка — считать, что раз поля нет в официальном списке требований, оно неважно: **`business_profile.url` в списке отсутствует, но именно он гасит больше аккаунтов, чем всё остальное вместе взятое** (см. ниже).

## Для приёма платежей (`charges_enabled`)

**Identity:**

| Поле | Примечание |
|---|---|
| `individual.first_name` | сверяется по базам, а не по документу — см. «Как проверяется identity» |
| `individual.last_name` | |
| `individual.dob` | |
| `individual.ssn_last_4` | **достаточно последних 4 цифр** |
| `individual.address` | для charges хватает **city + state + postal code** |

**Никакой документ загружать не нужно.** Ни в нормальном сценарии, ни на этапе сбора данных — см. ниже.

**Business:**

| Поле | Примечание |
|---|---|
| `business_profile.url` | ToS требует URL от всех, кто продаёт онлайн; сайт, соцпрофиль или приложение |
| `business_profile.product_description` | официально — замена сайта, если сайта нет; **сработала ли эта замена у нас — не проверено**, см. ниже |
| `business_profile.mcc` | индустрия |
| `business_profile.support_phone` | |
| `tos_acceptance.date` / `.ip` | ставит только сам пользователь через форму — платформе недоступно |

## Для выплат (`payouts_enabled`)

Всё вышеперечисленное **плюс**:

| Поле | Примечание |
|---|---|
| `external_account` | банковский счёт |
| `individual.address` | здесь уже **полный** адрес, а не city/state/ZIP |

> «Enabling payouts requires a validated full `identity.individual.address`. **Stripe disables payouts if we haven't verified the full address within 30 days.**»

**Charges и payouts независимы.** Принимать платежи без банка можно — деньги копятся на балансе Stripe, выплаты не уходят. На проде: 150 аккаунтов с `charges=on, payouts=off` (у 36.7% из них висит `company.tax_id` — пороги 1099), и 20 в обратном состоянии.

Наш гейт приёма платежей на `payouts` не смотрит: `IsDone = ChargesEnabled && DetailsSubmitted` (`StripeProvider.cs:76`).

## Как и когда это проверяется

**1. До принятия ToS не проверяется ничего.** Аккаунт инертен: `capabilities` = `inactive`, `disabled_reason=requirements.past_due`. Проверено: два тестовых аккаунта с заведомо проходной и заведомо провальной идентичностью **оба висели в `pending` 60+ секунд**, включая тестовый SSN `222222222`, который дока называет «immediate, returned directly in the response». Префилл через API сам по себе не даёт ни одного «yes» — он только укорачивает форму.

**2. Проверка запускается после сабмита формы** и идёт асинхронно. Замерено на живом онбординге: `details_submitted` → `charges_enabled` заняло **2 минуты 6 секунд**. Это и есть окно C1 — пользователь уже вернулся, а платежи ещё выключены.

**3. Драйвится вебхуком `account.updated`.** За одну сессию онбординга прилетает ~8 таких событий — по одному на шаг формы плюс переходы верификации. `previous_attributes` показывает, что именно изменилось.

**4. Как проверяется identity: по базам, а не по документу.** Никакой скан загружать не нужно — Stripe сверяет **введённые данные**:

> Stripe then attempts verification. In some cases, Stripe might be able to verify an account by **confirming some or all of the keyed-in data provided**. In other cases, Stripe might require additional information, including, for example, a scan of a valid government-issued ID, a proof of address document, or both.

То есть имя + DOB + `ssn_last_4` проверяются автоматически по источникам Stripe. Документ (`individual.verification.document`) — **фолбэк**: он лежит в `pending_verification` и переезжает в `currently_due`, только если сверка не сошлась. На проде затронул **163 аккаунта (0.8%)** `[прод]`, блокирует 118. Если не прошёл именно `ssn_last_4` — Stripe вместо документа может потребовать полный `id_number`.

**Практическое следствие — цена опечатки.** Провал сверки выглядит как код `verification_failed_keyed_identity` (152 случая на проде) и оборачивается запросом документа **позже, асинхронно**, когда пользователь давно ушёл. Поэтому опасен не отсутствующий документ, а неверно введённое имя — и это прямой аргумент против автоматического деления нашей единственной строки `Contacts.Name` на `first_name`/`last_name` по пробелу (см. [prefill-fields.md](prefill-fields.md), «Имя»): ошибка деления оплачивается не ошибкой на экране, а требованием фото через несколько дней.

⚠️ Отсюда же — почему экран S2 в макете дезинформирует: «A photo ID **or** SSN» читается как равноправный выбор, тогда как SSN спрашивают всегда, а фото — только у 0.8%, и только когда что-то не сошлось.

**5. Пороги добавляют требования позже.** По обороту (1099, `company.tax_id`), по регуляторике. Поэтому давно работающий мерчант может оказаться в состоянии «`details_submitted=true`, `charges_enabled=false`», внешне неотличимом от провала онбординга. Различают: `current_deadline` (стоят часы) и факт платежей (`GET /v1/charges?limit=1` с `Stripe-Account`).

## Сайт и описание: что требуется на самом деле

Правило Stripe: URL обязателен для всех, кто продаёт или продвигается онлайн — сайт, соцпрофиль или приложение. Сайта нет — вместо него идёт описание, но с требованием к содержанию:

> If the connected account doesn't operate a website… they're required to provide the `business_profile.product_description` instead. **A product description must detail the type of products being sold, as well as the manner in which the business charges its customers** (for example, in-person transactions).

То есть годное описание должно называть **и что продают, и как берут деньги**. «Handyman» этому не отвечает — а API такое принимает молча: валидация ≥10 символов живёт только в форме, не в API (проверено).

**Что показывает прод** `[прод]`:

| факт | значение |
|---|---|
| заполнили `url` среди начавших форму | 7.1% |
| соседние поля того же экрана: `mcc` / `product_description` | 50.9% / 52.4% |
| не включились, у кого `url` — единственный элемент `past_due` | **564** (2.6% от всех, кто нажал кнопку) |
| их медианный простой | 90 дней |
| контроль: подключённые с `url` в `past_due` | 7 из 3 323 |
| **подключённые, у кого `url` заполнен** | **3 310 из 3 323 (99.6%)** |

**Сайт у бизнесов есть** `[прод]` — последняя строка это доказывает. Посылка «сайта нет ни у кого» из [research.md](research.md) неверна: люди вписывают Facebook, Instagram или реальный сайт, когда их спрашивают. Верно лишь то, что **мы этих данных не храним**.

**Вывод** `[инференс]`: раз 99.6% дошедших сайт указывают, а 93% начавших его пропускают — поле надо спрашивать у себя, до хендоффа, а не полагаться на замену описанием. Для тех, у кого сайта действительно нет, второй рычаг — шаблон описания со способом оплаты.

⚠️ **Чего мы НЕ знаем.** Почему у этих 564 не сработала замена описанием — **не установлено**. Содержимое `product_description` через API не читается: после первого Account Link ключ исчезает из ответа `[тест]`. Известно лишь, что строка непустая (иначе поле висело бы в `currently_due`). Гипотеза «описания негодные» правдоподобна — API принимает и «Handyman» `[тест]` — но это **гипотеза**, а не замер. Разведёт её эксперимент на тестовой платформе: два аккаунта без сайта, у одного «Handyman», у другого текст по формуле Stripe.

## Две ловушки чтения requirements

**Ловушка №1 — базовый набор.** Голый аккаунт (`type=standard` + metadata, как создаёт `PreAuth`) с рождения должен шесть полей:

```
business_profile.product_description   business_profile.support_phone
business_profile.url                   external_account
tos_acceptance.date                    tos_acceptance.ip
```

Поэтому **«N% не выполнили поле X» ничего не доказывает**, если X в этом наборе: все должны его с нулевой секунды. Это распространённость, а не блокировка. На этом уже споткнулись дважды — уточнение в [research.md](research.md) и первая версия [prod-funnel.md](prod-funnel.md). Корректный тест: **единственность в `past_due`** + контроль на подключённых.

**Ловушка №2 — без capabilities требований почти нет.** Прод их не запрашивает, поэтому identity-требования не показываются, пока пользователь не начнёт форму. Зато это даёт самый надёжный маркер «не начинал»: `capabilities == {}` + базовая шестёрка = состояние побайтово равно свежесозданному.

## Чек-лист чтения состояния

1. `charges_enabled` → подключён.
2. `details_submitted=false` + `capabilities=={}` + базовая шестёрка → форму не открывали.
3. `details_submitted=false`, иное → бросил внутри формы; **где — по тому, чего в `currently_due` уже нет**, а не по тому, что там есть.
4. `details_submitted=true`, `charges_enabled=false` → смотреть **`past_due`** (не `currently_due`) и `disabled_reason`; **обязательно проверить наличие платежей**, иначе спутаешь «не включился» с «ограничен позже». На проде это 90/10 в пользу «не включился».
5. Любое «поле X блокирует N%» проверять на: входит ли X в базовый набор, единственное ли оно в `past_due`, и контроль на подключённых.
