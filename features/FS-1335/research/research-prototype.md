# FS-1335 — Локальный прототип: итерации и результаты (2026-07-03)

Локальный training-прототип (`C:\Git\_scratch\fs1335-price-model`, Python/TF CPU, данные из `invoicesapp-project-test:ml_training_us` (до 2026-07-06 — `fs1335_us`)). Три итерации модели против lookup-baseline'ов; протокол оценки — **cross-account холдаут** (10% аккаунтов по hash, ~8.7k аккаунтов, незнакомых модели), метрики в долларах после `exp(log-prediction)`.

Обучение: секунды на CPU (GPU не нужен — подтверждена гипотеза «Mercari-класс задач решается лёгкими моделями»).

## Итерации

| # | Формула | Данные |
|---|---|---|
| v1a | `log(p) = w[name] + w[state] + сезонность` (аддитивная) | 910k строк (словарь 6,075 имён: n≥30, accounts≥10; штат извлечён) |
| v1b | + `w[name×state]` (zero-init, L2-свип → 1e-5) | те же |
| v2 | + hashed word 1–2-grams (2^18, sum-pool, L2 1e-7, early stop) — Mercari-стиль | **3.71M строк** (все имена со штатом, без словарного фильтра) |

## Результаты

**Срез «показываемые» (имя в словаре, не suppress)** — то, что реально увидит пользователь v1:

| Модель | MAE | MdAPE | в ±25% |
|---|---:|---:|---:|
| v1a аддитивная | $383 | 0.555 | 24.7% |
| **v1b + interaction** | $379 | **0.488** | **30.7%** |
| v2 n-grams | $386 | 0.535 | 26.0% |
| lookup имя×штат | $378 | 0.500 | 30.3% |
| lookup имя | $377 | 0.504 | 28.2% |

**Срез OOV (имя не в словаре)** — 80% eval-строк, ранее «нет подсказки»:

| Модель | MAE | MdAPE | в ±25% |
|---|---:|---:|---:|
| **v2 n-grams** | **$608** | **0.881** | **15.2%** |
| lookup (откат на медиану имени/global) | $647 | 1.154 | 11.7% |

## Выводы

1. **v1b (exact name + state + регуляризованный interaction) — лучший кандидат для словарных имён**: единственная конфигурация, обошедшая lookup (MdAPE 0.488 vs 0.500). Interaction необходим: аддитивная модель выучивает state-эффект ≈ 0 (гео-сигнал живёт в ячейках имя×штат, а не в «общем уровне цен штата»).
2. **v2 n-grams — первый работающий OOV-прогноз**: MdAPE 0.88 против 1.15 у отката. Это открывает хвост (~47% трафика), но точность позволяет максимум «ориентир/диапазон», не точку. Без регуляризации и early stopping n-grams жёстко переобучаются на **аккаунт-специфичные формулировки** (train 0.86 vs val 1.38) — cross-account холдаут это ловит, обычный row-split не поймал бы.
3. На словарных именах v2 хуже v1b (0.535 vs 0.488) → **продакшн-форма = каскад**: имя в словаре → v1b-ветка; OOV → n-gram-ветка (с UX диапазона или консервативным suppress). Реализуемо и одним графом (обе ветки уже в v2-архитектуре), и двумя моделями.
4. Потолок точности класса задачи ≈ MdAPE 0.45–0.55 на словарных именах — согласуется с Mercari production-цифрой ±40% при большем наборе полей (см. `research-mercari.md`). Продукт должен обещать «пример цены», не «цену».

## Второй раунд (2026-07-05): каскад и альтернативы — все кандидаты замерены

Все пять кандидатов из чеклиста реализованы и прогнаны на том же cross-account холдауте (489,325 строк, 3.22M train).

### Каскад v1b+v2 (`eval_cascade.py`) — продакшн-форма, замерена end-to-end

Маршрутизация на холдауте: **v1b-ветка 13.3%** строк (словарь & !suppress), **v2/OOV 79.8%**, **молчание (suppress) 6.9%** → покрытие подсказкой 93.1%. (Замеренная доля словаря в потоке — ~20% строк, а не «~53% трафика» из ранних прикидок по head-именам.)

| Срез | cascade | lookup имя×штат |
|---|---:|---:|
| **Отвеченные (v1b+v2 вместе)** | MdAPE **0.830**, ±25% 17.4% | 0.978, 14.4% |
| Ветка v1b | 0.488 | 0.500 |
| Ветка v2 (OOV) | 0.881 | 1.154 |

Suppress-срез действительно ужасен у всех (lookup MdAPE 0.83–0.85 при MAE $1.27k) — молчание оправдано.

### Остальные кандидаты — сводка по OOV/shown срезам (MdAPE)

|---|---:|---:|---|
| v2 n-grams (базовая точка) | 0.535 | 0.881 | — |

На словаре по-прежнему хуже v1b → каскад сохраняется, меняется только OOV-ветка |

### Обновлённые выводы

1. **Продакшн-форма v1: каскад v1b + OOV-ветка + suppress-молчание** — подтверждена цифрами end-to-end (MdAPE 0.830 на отвеченных против 0.978 у lookup, покрытие 93%).
2. **Лучшая OOV-ветка — Model2Vec, а не хэшированные n-grams** (0.794 vs 0.881). Цена: артефакт potion-таблицы ~30 MB на устройстве (vs ~1 MB у хэша) + токенизация potion вместо md5-хэша в iOS-контракте. Замороженная uniq-таблица 1.36 GB из скрипта — тренировочное удобство; на устройство едет potion + голова.
3. **Квантильные головы — дешёвый и работающий апгрейд UX**: не теряют точку, дают честный сигнал «узко/широко» (узкий бакет вчетверо точнее среднего). Перед продом откалибровать ширину (расширить до coverage 50%, например температурой на pinball или post-hoc масштабом).
4. Табличный потолок достигнут: LightGBM с автоматическими взаимодействиями не бьёт v1b — сигнал (имя, штат, месяц) исчерпан, дальнейший рост только из текста (что Model2Vec и показал).

## Третий раунд (2026-07-05): улучшения по мотивам web-обзора 2026

Три кандидата из обзора современных практик (conformal prediction, hierarchical pooling, embedding retrieval a-la Airbnb), все замерены против golden-set.

### CQR-калибровка интервалов (`calibrate_quantiles.py`) — ✅ работает

Split-conformal поправка (Romano et al.) поверх обученных квантильных голов, без переобучения: калибровочная половина holdout-аккаунтов даёт поправку Q=+0.195 log (границы ×1.215), проверка на второй половине:

| Срез | Coverage до | Coverage после (цель 50%) |
|---|---:|---:|
| all | 43.7% | **53.2%** |
| shown | 44.5% | 58.6% |
| OOV | 43.3% | **52.1%** |

Перелёт на shown (58.6%) — поправка глобальная; следующий шаг при желании — Mondrian/per-slice CQR (отдельный Q для shown/OOV). Бакет «точка» после калибровки на shown имеет coverage 97.8% при доле 4.2%.

### v1c: имя×регион (partial pooling) — ❌ в текущем виде отклонён

Гипотеза: ячейка имя×Census-регион (~10× больше данных) как промежуточный уровень между штатом и страной вылечит шумные interaction-ячейки. Результат двоякий, в сумме — нет:

- Гео-ordering усилился (mowing ratio 1.39→1.70, handyman 1.43→1.77, junk 1.25→1.61, все PASS) — региональное направление модель ловит лучше;
- НО rank corr по штатам упал 0.24→0.16, strict in-range mowing 11/24→6/24, и **точность точки на shown деградировала 0.488→0.519**. Свип выбрал слабую регуляризацию региона (1e-5), которая перетянула сигнал из точных ячеек.

Вывод: идея не закрыта (нужна двухступенчатая подгонка residual'ов или жёсткий приор), но quick-версия хуже v1b — остаёмся на v1b. Артефакты в `artifacts_v1c/`.

### kNN-retrieval OOV-ветка (`knn_retrieval.py`) — ✅ сюрприз раунда

Без обучения вообще: potion-эмбеддинг фразы → cosine top-10 среди 6,075 словарных имён (не-suppress, порог sim 0.35) → взвешенная медиана их v1b-предсказаний.

| OOV-ветка | 35 внешних проб в диапазоне | Holdout OOV (MdAPE) |
|---|---:|---:|
| **kNN retrieval** | **97.1%** (34/35) | 0.834 (30k-сэмпл) |
| v3 Model2Vec | 80.0% | **0.794** |
| v2 n-grams | 51.4% | 0.881 |

Интерпретация: на **реалистичных перефразировках частых услуг** (а это и есть типичный OOV нового пользователя) kNN почти безошибочен, потому что наследует откалиброванные v1b-статистики соседа; на **полном хвосте** (16.6k уникальных имён сэмпла, много мусора без близких соседей) выигрывает обученная v3-голова. Бонус kNN — объяснимость: top-1 сосед («grass cutting service» → «grass cutting», sim 0.94) можно показывать в UX как «похоже на X».

**Обновлённая продакшн-форма OOV-ветки — гибрид**: сосед ближе порога (sim ≥ ~0.7) → kNN-ответ с объяснением; иначе → v3-голова; совсем далеко → молчание/широкий диапазон. Обе ветки едут на одних и тех же potion-эмбеддингах — на устройстве это одна таблица.

## Четвёртый раунд (2026-07-06, при переносе в пайплайн): урок про сиды и финальный рецепт v1b

При прогоне пайплайна `ml/price_model` на свежих prod-данных гейт завалил `beats_lookup` (p50 0.52 против lookup 0.500), хотя «вчера» было 0.49. Свипы по сидам вскрыли неприятное: **у v1b ±0.03 MdAPE шума по сидам** (0.47–0.53), и «эталонные 0.488/0.490» были удачей сида 1335 — в прототипе это молча абсорбировал L2-свип (лучший из 5 прогонов по val), который при упрощении в пайплайн был выброшен. Дополнительно: квантильные головы, обучаемые pinball'ом с нуля, стоят p50 ещё ~0.02 (градиент q=0.5 — полу-масштабный MAE + три цели на общих эмбеддингах); warm-start с файнтюном возвращает лишь половину.

**Финальный рецепт v1b (в пайплайне):**
1. Точечная модель, рецепт прототипа (MAE, lr 0.05, 8 эпох) × **5 рестартов**, отбор по чистому val MAE(log) — возвращает механизм отбора, съедающий сид-шум. Свип 3 диаг-сидов: 0.473/0.493/0.488 — все бьют lookup.
2. Квантили = **замороженная точечная база + обучаемые offset-головы p25/p75** (zero-init, dim-2 члены name/state/cell/seasonal): p50 = точечный оптимум по построению и деградировать не может; pinball учит только раскрытие интервала. Coverage до CQR ~37%, после CQR 46.9% (в допуске).

Итог на prod-данных (run 20260706_smoke2): **GATE PASS, shown p50 MdAPE 0.473 — лучший результат фичи** (прототип 0.488), external 93.0%, гео 4/4 (ratio 1.25–1.38), kNN-пробы 91.4%. Попутно гейт отработал боевым образом дважды: поймал шумный T3-ярус state-регэкспа в новых SQL-рутинах (см. `build_price_training_rows.sql`) и сид-нестабильность рецепта.
