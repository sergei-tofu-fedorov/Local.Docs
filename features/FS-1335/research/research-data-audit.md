# FS-1335 — Аудит данных: хватает ли их для модели «название айтема + гео → цена»

Аудит prod BigQuery `inv-project.ai_analysis_us` (2026-07-03, read-only, ~25 GB scanned ≈ $0.16). Вопрос: есть ли данные для датасета «название позиции + гео клиента → unit price» и в каком объёме/качестве.

**Вердикт: данные есть, задача осуществима.** Рабочий датасет после всех фильтров — **5.48M line items (USD, с адресом клиента, 2018+)**; без валютного фильтра — 9.5M. Главные препятствия: адреса — сырой свободный текст (нужен геокодинг → geohash), и тяжёлый хвост названий (словарь покрывает ~30–53%, хвосту нужен текстовый fallback).

| Порог частоты | Названий | Покрытие строк |

| ≥100 | 4,314 | 1.66M (30%) |

Топ названий — generic-услуги: `labor` (237k, 43k аккаунтов, медиана $150), `materials`, `delivery`, `shipping`, `cleaning`, `service call`, `mowing`, `lawn maintenance`… Встречаются и аккаунт-специфичные (`varsity game fee` — 45 аккаунтов; `picada de res` — 6 аккаунтов): частотность ≠ универсальность, надо смотреть и на число аккаунтов.

Вывод для энкодинга: словарь по голове (~5k–50k названий) покрывает 30–53% трафика; для хвоста нужен fallback (текстовые фичи: слова/char n-grams через hashing — конвертируемо в Core ML, т.к. индексы integer; или каскад «словарь → гео-среднее»).

## Гео: адреса — сырой текст, геокодинг обязателен

`src_clients.info[].address` — одна строка свободного текста. Выборка показывает весь спектр:
- полные US-адреса: `7818 Switchgrass Creek Drive, Katy, TX 77493` — геокодируются легко;
- частичные: `1020 Wozencraft`, `Stanford`, `767 Knox Ave S Lehigh Acres` — city/state угадываются не всегда;
- международные: UK (`… Sutton Coldfield, B72 1PA`), Dubai, Коста-Рика, урду-вязь;
- мусор: `Bhh`, `N/A`, `GST : 362830AHHGR543`.

Реального «lat/lon» в данных нет нигде; `dim_account.state` пуст. Значит, шаг геокодинга (или хотя бы извлечение state/ZIP регэкспом для US) — обязательная часть dataset job. ZIP→geohash/центроид — дешёвый первый приближённый вариант без внешнего геокодера.

## Гочи для dataset job (проверено на живых данных)

1. **Композитный ключ клиента**: `src_clients.id` = `<account_id>|<client_guid>`, а `src_invoices.client_id` — голый guid, **местами с дефисами, местами без**. Рабочий join: `REPLACE(LOWER(src_invoices.client_id),'-','') = REPLACE(LOWER(SPLIT(src_clients.id,'|')[SAFE_OFFSET(1)]),'-','')` + равенство `account_id`. Наивный `client_id = id` даёт 0 совпадений.
2. `dim_account.state` — пустой у всех строк; `trade` — только 11k аккаунтов. На account-гео рассчитывать нельзя.
3. Даты содержат мусор (год 1…9627) — фильтровать окно явно.
4. У клиента `info` — массив; адресов может быть несколько (брать первый непустой / последний?).
5. 14% инвойсов без `currency_code` — решить: выкидывать или вменять по гео.
6. `mart_invoice_line_items` не содержит валюту и клиента — обязательный join к `src_invoices`.

## Глобальный план (фазы)

0. **Scope-решения** (до кода): ~~только USD/US в v1?~~ **решено 2026-07-03 — только US** (фильтр USD + US-адрес клиента); определение unit price (поле `unit_price` есть — ok); гейт «новый пользователь» — вне модели (клиентская логика).
1. **Dataset job** (`Tofu.AI.Backend`, BQ → training table): фильтры воронки выше + нормализация названий + гео-шаг (v1: state/ZIP-извлечение регэкспом → ZIP-центроид → geohash-4/5; v2: полноценный геокодер) + словарь `name→int` (порог по частоте и числу аккаунтов) + анти-выбросы по цене. Materialized-таблица в `ai_analysis_us` (или отдельный ML-датасет).
2. **Baseline без ML** (важно для оценки смысла): предсказание = медиана цены по (название-из-словаря × geohash-префикс) с откатом на более короткий префикс → глобальную медиану. Метрика MAE. Любая модель обязана бить этот lookup — если не бьёт, модель не нужна.
3. **Линейная модель (TF)**: фичи — словарь названий (int-индексы, multi-hot/эмбеддинг малого ранга), geohash target-encoding с префиксным fallback, sin/cos месяца; таргет — log(unit_price) скорее всего (цены лог-нормальны). Валидация: time-based walk-forward, срезы по регионам и частотным бакетам названий (голова vs хвост).
4. **Vertex AI пайплайн**: CustomJob (prebuilt TF, CPU) → экспорт → конвертация `coremltools` → parity-gate TF↔Core ML → Model Registry; retraining по cron. Обучение — только `invoicesapp-project-test` (данные придётся реплицировать/читать из prod BQ — решить механизм доступа).
5. **Дистрибуция**: manifest endpoint + GCS/CDN (по web-spike), артефакт = `.mlpackage` + словарь `name→int` + geohash-словарь, версионируются вместе.
6. **Консьюмер/iOS**: контракт входов модели, on-device нормализация названия (должна быть бит-в-бит с обучением — зафиксировать спеку), fallback UX когда предсказания нет.

Порядок принципиален: фаза 2 (baseline) — дешёвый go/no-go до инвестиций в Vertex-инфру.

## Замер 2026-07-03: извлекаемость гео регэкспом (на `src_price_line_items`, 5.48M строк)

| Ярус | Строк | % |
|---|---:|---:|
| T1: код штата непосредственно перед ZIP (высокая уверенность) | 2.99M | 54.5% |
| T2: любой standalone код штата (последнее вхождение) | 3.72M | 67.8% |
| T3: T2 + полное название штата («Texas», …) | 4.03M | 73.6% |
| ZIP (последняя 5-значная группа) | 3.63M | 66.2% |
| Любой гео-сигнал (штат или ZIP) | **4.28M** | **78.1%** |
| Ничего не извлечь | 1.20M | 21.9% |

**Вывод: внешний геокодер в v1 не нужен.** Регэксп-извлечение даёт state-уровень для ~2/3–3/4 строк; ZIP→центроид→geohash покрывает 66%. Строки без гео-сигнала (22%, среди них не-US адреса из USD-инвойсов и мусор) просто выпадают из обучения — остаётся ~4.3M строк, этого достаточно. Геокодер — опция v2 для повышения точности/покрытия.

Оговорки: коды с ложными срабатываниями (`CT`=Court, `LA`, `IN`, `OR` как слова) — T1-ярус от них свободен, T2 слегка завышен; «WASHINGTON» ловит и улицы. Для v1 предлагается: T1 → ZIP-префикс → T2/T3 → drop.

## Замер 2026-07-03: дисперсия цены внутри (название × штат)

Фильтр выбросов `0 < unit_price ≤ 50000`, группы с n≥30, квартильные метрики (rel IQR = (p75−p25)/медиана):

| Скоуп | Групп | Строк покрыто | Медианный rel IQR | Медианный p75/p25 |
|---|---:|---:|---:|---:|
| название | 16,773 | 2.29M | 0.32 | 1.36 |
| название × штат | 11,703 | 1.18M | **0.20** | 1.22 |

**Вывод: сигнал есть, feasibility подтверждается.** У типичного названия цена кучная (половина массы в ±16–18% от медианы), штат дополнительно сужает разброс в ~1.6 раза. Гео-сигнал реален: медианы по штатам для одного названия расходятся в 2–4 раза (p90/p10 по топ-названиям).

Но по **generic-названиям из головы словаря** разброс огромен и штат его почти не спасает: `labor` rel IQR 3.27 (по штатам 3.03), `materials` 2.77, `paint` 2.75, `plumbing` 2.69 — это «вёдра» разнородных работ, для них точечная «средняя цена» малоинформативна. Умеренные head-названия ведут себя хорошо: `mowing` 0.75, `service call` 0.86, `lawn maintenance` 0.94, `rent` 0.91; у сервисных названий штат заметно помогает (`lawn service` 1.42→0.62, `transportation` 1.17→0.56, `cleaning` 1.13→0.73).

Следствия для дизайна:
- generic-«вёдра» (labor, materials, parts…) — кандидаты на подавление подсказки или широкий интервал вместо точки; полезен выход модели с оценкой уверенности (квантильные головы p25/p50/p75 вместо одной точки).
- аномальные p90/p10 у `delivery` (33×) и `transportation` (25×) — семантическая неоднородность (доставка пиццы vs грузоперевозка) — лечится только контекстом аккаунта/категории, не гео.
- ⚠️ методологическая оговорка для валидации: кучность имён частично объясняется тем, что редкое название живёт внутри 1–2 аккаунтов с фиксированным прайсом. Для целевой аудитории («новые пользователи») валидация обязана быть **cross-account** (holdout по аккаунтам, не только по времени), иначе метрики будут льстить.

### Обращение с generic-«вёдрами»: варианты и цена (замер 2026-07-03)

Доля трафика по разбросу имени (имена n≥30; 58% строк — хвост n<30, он живёт на fallback-механике и сюда не входит):

| Бакет rel IQR | Имён | Строк | % трафика |
|---|---:|---:|---:|
| ≤ 0.5 (кучные) | 9,633 | 988k | 18.1% |
| 0.5–1.0 | 2,320 | 361k | 6.6% |
| 1.0–1.5 | 1,473 | 236k | 4.3% |
| 1.5–2.0 | 1,049 | 170k | 3.1% |
| > 2.0 (generic-вёдра) | 2,298 | 532k | **9.7%** |

Варианты:
1. **Из обучения не удалять** — в модели с per-name весами (multi-hot/lookup) generic-имя получает свой вес и чужие не портит, а его строки полезны для гео-энкодинга. «Удаление» нужно только на serving-стороне.
2. **Data-driven suppress-флаг по разбросу (рекомендация v1):** dataset job считает per-name (и name×state) rel IQR; имена выше порога получают флаг в словаре артефакта (словарь и так версионируется с моделью). Порог 2.0 давит 9.7% head-трафика, 1.5 — 12.8%. Автообновляется при retraining, без ручной курации, языконезависимо.
3. **Ручной blocklist** — быстро, но рукопашно и стареет; годится лишь как надстройка над (2) для топ-имён.
4. **Квантильный выход модели (рекомендация, если успеем в v1):** предсказывать p25/p50/p75 (pinball loss, уже в web-spike); клиент показывает точку при узком интервале, диапазон при среднем, ничего при широком. Generic-вёдра давятся автоматически, решение выносится из словаря в саму модель.
5. **Дизамбигуация контекстом (v2):** разбивать «labor» по trade/соседним позициям — для новых пользователей контекста мало, в v1 не тянем.

**Решено 2026-07-03: принята комбинация 1 + 2 (+ 4 как upgrade)** — обучаемся на всём, словарь несёт `suppress`-флаг (порог rel IQR ≈ 2.0) и per-name статистики, квантильные головы — апгрейд v1.
