# FS-1335 — Разбор Mercari Price Suggestion Challenge (каноничный аналог задачи)

Проверка первоисточников по Kaggle-соревнованию Mercari (2017–2018): предсказание цены товара по свободнотекстовому названию/описанию. Ближайший публичный аналог FS-1335 — что реально сработало у победителей и что из этого переносится на нашу подсказку цены. Проверено 2026-07-03 по репозиторию победителей, отчёту Mercari Engineering и решению 25-го места.

## Задача и данные (у них)

- ~1.48M проданных позиций из US-приложения Mercari; поля: `name`, `item_description` (свободный текст), `category`, `brand`, `item_condition`, `shipping`.
- Таргет — цена продажи; метрика **RMSLE** (≡ RMSE на log-цене) — тот же выбор «ошибки в относительных величинах», что наш log-target + MAE.
- Уникальное ограничение: **kernel-only** — всё решение обязано обучиться и предсказать в Kaggle-кернеле (~1 час, слабый CPU/память). Из-за этого выиграли лёгкие модели, а не тяжёлый DL.

## Что сработало (проверенные факты)

**1-е место — Paweł Jankiewicz & Konstantin Lopuhin, RMSLE 0.3733** ([репо решения](https://github.com/pjankiewicz/mercari-solution)):
- **Разреженные текстовые фичи**: TF-IDF и бинарные count-векторы по word n-grams, конкатенация текстовых полей; никакого тяжёлого NLP.
- **Малые MLP поверх sparse-матрицы** (знаменитый упрощённый вариант — [«Mercari Golf»: 75 строк, RMSLE 0.3875](https://www.kaggle.com/code/lopuhin/mercari-golf-0-3875-cv-in-75-loc-1900-s)) — по сути «чуть-нелинейная» линейная модель на n-gram признаках.
- Ансамбль из 8 вариантов (Huber-регрессия / классификация по бинам цены × варианты бинаризации данных), взвешенное усреднение.
- RNN/CNN-эмбеддинги **не окупились** в рамках compute-бюджета — sparse-подход их обошёл.

**Отчёт Mercari Engineering** ([blog](https://engineering.mercari.com/en/blog/entry/2018-11-14-172509/)):
- Интерпретация точности: RMSLE 0.3875 ≈ для товара за 3000 JPY предсказание живёт в полосе **2051–4387 JPY (примерно ±40%)** — и Mercari называет это «acceptable for production use».
- Перенос победной модели на японское приложение **не воспроизвёл** score — доменный сдвиг реален, модель надо учить на своём рынке/языке.

**25-е место (Ahmet Erdem, [репо](https://github.com/aerdem4/mercari-price-suggestion))**: ансамбль FM_FTRL (факторизационная машина на n-grams) + NN с word-эмбеддингами — подтверждает общий паттерн «sparse text + лёгкая модель».

> Поправки к устному пересказу в чате: 0.3875 — это score *упрощённого 75-строчного кернела*, полное победное решение — **0.3733**; репозиторий aerdem4, который поиск подсунул как «winning approach», — на самом деле **25-е место** (сам подход при этом типовой для топа).

## Что это значит для FS-1335

1. **Класс точности совпадает.** Их «production-acceptable» ±40% при *шести* полях — наш прототип с *двумя* признаками (имя+штат) даёт MdAPE 0.49. Мы в той же лиге, и ждать «±10%» от этой задачи не стоит ни при какой модели — продуктовое ожидание должно быть «ориентир/диапазон», не «точная цена».
2. **Главный неиспользованный резерв — текст внутри названия.** Наш словарь точных имён — вырожденный BoW, где всё название = один токен: «lawn mowing front yard» и «mowing» не делят статистику. Word n-grams (1–2) поверх названий — прямой кандидат v2; линейная модель на sparse-фичах остаётся конвертируемой (гейт: hashing/токенизация должны выполняться на устройстве бит-в-бит со спекой обучения — усложнение контракта, из-за которого это и отложено из v1).
3. **Лёгкие модели — не компромисс, а оптимум** для этого класса задач: победитель уложился в 1 CPU-час. Наши секунды обучения на ноутбуке — нормальный масштаб, Vertex нужен для воспроизводимости/расписания, не для мощности.
4. **Huber/классификация-по-бинам** — приёмы устойчивости к выбросам сверх log-target; бины цены (клас-фикация) — ещё один способ получить «диапазон» для UX помимо квантильных голов.
5. **Доменный сдвиг**: чужие пре-обученные ценовые модели/датасеты нам не помогут — учим на своих инвойсах (что мы и делаем). Также осторожно с переносом между сегментами внутри нас самих (штат/трейд).

## Актуальность в 2026 (соревнование 2017–2018 — не устарело ли?)

Проверено 2026-07-05 по свежим источникам. Фронтир моделирования ушёл вперёд, но уроки Mercari для FS-1335 остаются в силе — по конкретной причине: **наши ограничения воспроизводят их ограничения**.

Что изменилось:

- SOTA для text-to-price сегодня — файнтюненные LLM: [Quantile Regression with LLMs for Price Prediction (arXiv 2506.06657, 2025)](https://arxiv.org/abs/2506.06657) — Mistral-7B с квантильными головами обходит энкодеры, embedding-подходы и few-shot. Попутно валидирует нашу идею квантильных голов для «диапазона».
- Semantic-эмбеддинги сильно обошли TF-IDF на регрессии по тексту: contextual-эмбеддинги дают Ridge R² ~0.81 против ~0.44 у TF-IDF 1–2-grams ([бенчмарк](https://machinelearningmastery.com/llm-embeddings-vs-tf-idf-vs-bag-of-words-which-works-better-in-scikit-learn/)) — за счёт синонимии, а не совпадения токенов.
- Для табличных фич стандарт по-прежнему GBDT (LightGBM/CatBoost) — LightGBM-бенчмарк из `research-prototype.md` валиден.

Почему рецепт Mercari всё равно оптимум для нас:

1. Их kernel-only лимит (1 CPU-час) ≈ наш лимит **on-device Core ML на iPhone** (миллисекундная подсказка при вводе строки). 7B-LLM на устройство не влезает, серверный inference ломает архитектуру v1 (латентность/приватность/стоимость). В классе «маленькая модель на устройстве» sparse+light — по-прежнему оптимум, не устаревший компромисс.
2. Потолок точности задан шумом данных, не моделью: дисперсия внутри (имя×штат) неустранима — большая модель не извлечёт информацию, которой нет во входе из двух признаков. «±40% = production-acceptable» остаётся правильной продуктовой калибровкой.
3. Log-target, cross-account валидация, доменный сдвиг — методология, не зависящая от эпохи.

Что современный ML реально может дать (кандидат v3, см. `research-prototype.md`): маленький дистиллированный энкодер (класс MiniLM, ~22M параметров, конвертируется в Core ML) как фичер OOV-ветки — единственное место, где semantic-эмбеддинги («grass cutting» ≈ «lawn mowing») обещают большой выигрыш над хэшированными n-grams.
