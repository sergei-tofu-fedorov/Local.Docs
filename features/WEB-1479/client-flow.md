# WEB-1479 — Клиентский флоу: проброс авторизации iOS → Web (сценарий, краевые случаи, контракты BFF)

Компактный план для клиентских команд (iOS, Web): как работает проброс авторизации, какие у BFF контракты и какие краевые случаи нужно обработать. Решения и источники — в [README](./README.md) и [web-spike](./web-spike.md). Значения с пометкой *(предварительно)* финализируются на `/plan write`.

Терминология: **handoff-токен** — одноразовый opaque-токен переноса сессии (аналог Auth0 Session Transfer Token); **custom token** — Firebase-токен, которым реально логинится веб. Handoff-токен сам по себе не аутентифицирует — он только обменивается на custom token.

## Основной сценарий

Роутинг финальных страниц живёт **на стороне iOS**: приложение само выбирает, какую страницу веб-аппа открыть, и дописывает handoff-токен во фрагмент. Сервер страницу не возвращает и о ней не знает (это снимает открытый вопрос README «где хранится роутинг переходов»).

```
1. iOS (пользователь залогинен, есть Firebase JWT) — тап «открыть веб»
2. iOS → POST /api/authenticate/handoff-tokens (Bearer JWT)
   ◄─ { token }                   токен одноразовый, TTL 60–120 с (предварительно)
3. iOS выбирает целевую страницу и открывает внешний браузер:
      https://app.tofu.com/<выбранный-путь>#t=<token>     (токен во ФРАГМЕНТЕ, не в query)
4. Web (бутстрап приложения, на ЛЮБОЙ странице): синхронно читает hash →
   history.replaceState (стирает токен из URL)
5. Web → POST /api/authenticate/handoff-tokens/exchange { token }   (аноним, HTTPS)
   ◄─ { customToken }
6. Web: firebase.auth().signInWithCustomToken(customToken) → остаётся на открытой странице
```

## Контракты BFF

Ответы — в стандартном конверте BFF: успех `{ "result": { … } }`, ошибка `{ "error": { "code", "message", "info", "traceId" } }`.

### `POST /api/authenticate/handoff-tokens` — выпуск токена

Требует `Authorization: Bearer <Firebase JWT>` (как остальные авторизованные вызовы BFF).
Тело запроса: пустое. Целевую страницу сервер не возвращает — её выбирает iOS.

```jsonc
// 200
{ "result": {
    "token": "tht_Qx7…",           // одноразовый opaque handoff-токен: tht_ + Base64Url(32 байта)
    "expiresInSeconds": 120        // предварительно — чтобы клиент не открывал заведомо мёртвый токен
} }
```

Ошибки: стандартные авторизационные (`unauthorized`, `token_expired`, `token_revoked`, `missing_claims`) → как во всех авторизованных эндпоинтах; ничего handoff-специфичного.


### `POST /api/authenticate/handoff-tokens/exchange` — обмен токена

Анонимный. Вызывается только веб-аппом. Токен передаётся **в теле запроса** (не в path и не в query — секреты в URL попадают в access-логи).

```jsonc
// запрос
{ "token": "tht_Qx7…" }

// 200
{ "result": { "customToken": "<Firebase custom token>" } }
```

Ошибки (`error.code`):

| `error.code` | Когда | Действие клиента (web) |
|---|---|---|
| `handoff_token_used` | токен уже погашен (повторный заход, bfcache, дубль-запрос) | есть Firebase-сессия → остаться на странице; нет → expired-UX |
| `handoff_token_expired` | TTL истёк | expired-UX: «вернитесь в приложение и нажмите ещё раз» |
| `invalid_handoff_token` | токен не найден / мусор | expired-UX (та же страница, без деталей) |

> HTTP-статус ошибок обмена: на стороне Tofu.Auth — 410 Gone (паттерн magic-login); прокинет ли BFF 410 или отдаст 400 с тем же `error.code` — решение на `/plan write`. **Контракт для веба — строка `error.code`, не статус.**

## iOS: порядок работы и краевые случаи

- **Роутинг — ответственность iOS.** Приложение хранит карту «куда ведёт кнопка → web-путь» и строит URL само: `"https://app.tofu.com/<путь>#t=\(token)"`. На этапе 1 — главная страница; новые цели добавляются без изменений бэкенда.
- **Новый токен на каждый тап.** Токен запрашивается непосредственно **перед** `open(url)` (не при открытии экрана) — TTL короткий, холодный старт браузера его съедает. Токены не кэшировать и не переиспользовать.
- **Двойной тап** — два валидных токена, два таба; оба обмена пройдут. Это нормально, ничего не блокировать.
- **Токен — во фрагменте**, не в query (утечки через логи/Referer — см. README). Если целевой URL уже содержит свой фрагмент/якорь — не комбинировать; такие цели не поддерживаем.
- **Каждый целевой путь обязан удовлетворять двум условиям** (требование ко *всем* открываемым страницам, не к одному лендингу):
  1. не отдаёт server-side redirect (редирект наследует фрагмент — токен уедет в новый URL);
  2. не перехватывается universal links (иначе iOS вернёт URL в приложение — петля). Проверить AASA на все целевые пути до старта работ; тестировать на устройстве.
- **«Safari» = браузер по умолчанию.** `UIApplication.open(https:)` откроет Chrome/Firefox, если пользователь сменил дефолт. Флоу работает так же, но долгоживущая веб-сессия осядет в том браузере. В QA-матрицу — кейс «Chrome по умолчанию».
- **Ошибка выпуска токена** (401 и т.п.) — обычный ре-логин-флоу приложения; браузер не открывать.

## Web: порядок работы и краевые случаи

Выделенного лендинга нет — обработка токена живёт в **бутстрапе приложения** и срабатывает на любой странице, открытой с `#t=<token>`. Машина состояний:

```
hash содержит #t=<token>:
  1. прочитать токен СИНХРОННО (до инициализации роутера), в переменную
  2. history.replaceState — стереть фрагмент СРАЗУ (до любого await),
     сохранив path/query открытой страницы
  3. exchange → signInWithCustomToken → остаёмся на открытой странице
     - «свежий токен всегда выигрывает»: подписываем даже при живой сессии
       (аккаунт в приложении мог смениться; signInWithCustomToken заменит юзера)
  4. exchange упал:
     - handoff_token_used → есть сессия ? остаёмся на странице : expired-UX
     - expired / invalid  → есть сессия ? остаёмся : expired-UX
hash пуст:
  обычный флоу приложения (токен не участвует)
```

- **Порядок в бутстрапе:** обработка `#t=` должна отработать **до** того, как приложение решит «не залогинен → на /login» — иначе редирект на логин потеряет контекст (и, как любой клиентский redirect до зачистки, может утащить фрагмент).
- **Once-guard на обмен** — React 18 StrictMode монтирует эффекты дважды → второй exchange получит `handoff_token_used`. Guard через ref/модульный флаг, не по содержимому hash.
- **bfcache:** Back→Forward может восстановить страницу с исходным URL (кейс Supabase auth-js #302) — повторный заход обязан корректно проходить ветку `handoff_token_used`.
- **Целевые страницы без редиректов.** Любой server-side redirect (`/path`→`/path/`, http→https-цепочки, CDN) наследует фрагмент — токен уедет в новый URL. Прямая выдача всех страниц, которые открывает iOS; `Referrer-Policy: no-referrer`.
- **Зачистка телеметрии — глобально, вместе с обработчиком.** Sentry JS пишет `location.href` с фрагментом в события и breadcrumbs — `beforeSend` + `beforeBreadcrumb` обязаны вырезать токен до первой продакшен-ошибки. Благодаря префиксу зачистка — точный regex `tht_[A-Za-z0-9_-]+` (ловит токен и в URL, и в breadcrumbs, и в текстах сообщений), а не «всё после `#t=`». Раз токен может прийти на любую страницу, зачистка ставится на уровне всего приложения. То же для любой аналитики, собирающей полный URL.
- **Токен нигде не хранить** (никаких localStorage/sessionStorage) — только переменная на время обмена.
- **Кросс-таб:** Firebase-сессия общая на origin (IndexedDB). Повторный handoff во втором табе сменит юзера и в первом — приложение должно реагировать на `onAuthStateChanged` (как минимум перезагрузить данные).

## Повторное открытие — два правила

1. **Свежий токен всегда выигрывает.** Каждый тап в приложении → новый токен → web безусловно обменивает и заменяет текущую веб-сессию. Это единственный путь, корректно обрабатывающий смену аккаунта в приложении.
2. **Повтор того же URL (погашенный токен) — мягкий вход.** `handoff_token_used` + живая сессия = успех (продолжаем); без сессии = expired-UX. Какому uid принадлежал погашенный токен, сервер не раскрывает — устаревшая ссылка после смены аккаунта продолжит старую сессию; это принятый компромисс (правило 1 — корректирующий путь).
