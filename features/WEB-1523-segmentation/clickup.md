# WEB-1523 — дерево задач (Phase 1 / Phase 2)

Дерево для разнесения по ClickUp. Phase 1 = «выкатить прототип на прод быстро»: только FSM-fit, без обучения, без сложных инструментов. Phase 2 = всё, что делает анализ *лучше со временем* или добавляет новое.

**Масштаб v1:** ~50 000 подходящих аккаунтов (старше 3 месяцев + есть активность по invoice/estimate за последний месяц). При таком объёме простой подход — **Hangfire + чтения через Mongo Data Federation поверх снапшотов** — нам подходит. Прод Mongo при этом не нагружаем: запросы уходят на федеративный endpoint, а не на живые кластера. Стейдж-окружение проще: там можно ходить напрямую в реальные прод-кластера Mongo обычным connection string (как остальные сервисы) — имена коллекций совпадают, отдельная Data Federation на стейдже не нужна. Более тяжёлые варианты (полноценная копия в BigQuery, read-replica, отдельный планировщик) оставляем на Phase 2, если решим расти.

## Phase 1 — прототип на прод

- ✅ **SPIKE Сегментация пользователей**
- 🔵 **Выборка для валидации FSM-fit**
- 🔴 **Расширение Tofu.AI.Backend Infrastructure** *(оттестировано на stage, нужен cleanup перед review)*
  - Клиент BigQuery, Storage Write API, сайдкары Presidio, клиент OpenAI, service accounts
- 🆕 **Зафиксировать v1-сценарий PM/маркетинга + контракт результата в BigQuery** — **4h** *(можно делать параллельно с имплементацией)*
  - Один конкретный сценарий от PM (например: маркетинг таргетит аккаунты с `fsm_fit.tier = high`)
  - Какие поля и в каком виде PM читает из BigQuery → это задаёт структуру запроса в LLM и колонок таблицы
- 🆕 **Сбор метрик для анализа** — **8h**
  - Откуда берём данные: **Mongo Data Federation поверх снапшотов** (решение зафиксировано — direct-reads / Analytics Node в Phase 1 не идём). Worker подключается к федеративному endpoint, прод-кластера не нагружает.
  - **Снапшоты — нужен `clients`.** Сейчас снапшоты сделаны не для всех нужных коллекций; для FSM-fit нужен ещё `clients`. В рамках этой задачи: (а) посмотреть существующую структуру снапшотов (какие коллекции уже снапшотятся, расписание, формат, какой `_id`/partition layout), (б) по этому же шаблону добавить `clients`, (в) убедиться что `invoices` и `estimates` снапшоты покрывают окно, нужное метрикам (30d / 12mo per [`analyses/metrics.md`](analyses/metrics.md)).
  - **Стейдж-окружение:** Data Federation на stage **не поднимаем** — Worker на stage ходит напрямую в реальные прод-кластера Mongo обычным connection string (как остальные сервисы). Имена коллекций (`invoices`, `estimates`, `clients`, `accounts`) совпадают между прод и нашим федеративным датасетом, так что код запросов один и тот же; различается только connection string по окружению. Это упрощает stage-прогон и end-to-end тест без отдельной федерации.
  - Кто запускает: Hangfire подходит для такого объёма, оставляем. Альтернативы (Cloud Run Jobs / Workflows / Pub-Sub) — Phase 2
  - Параметры пакета: сколько аккаунтов за один прогон, как часто (раз в сутки или чаще), в какое окно (ночью)
  - Кого берём в выборку: только подходящие аккаунты (старше 3 месяцев + активность по invoice/estimate за последний месяц), ~50k на v1
  - Механизм «что изменилось — нужно пересчитать» (`input_hash` от метрик + версий)
- 🆕 **LLM-анализ FSM-fit и запись результатов** — **8h**
  - Один промт, зашитый в код (`prompt_version = 1`)
  - Правила в коде: `Analyses.Domain/Rules/FsmFitRule.cs`, версия — через колонку `rule_version`
  - На уровне записи храним версии: `schema_version`, `rule_version`, `prompt_version`, `model_id`
  - Запись в BigQuery через Storage Write API (UPSERT в режиме CDC — одна строка на аккаунт, перезаписывается при пересчёте)
  - Режим прогона без записи: флаг `--dry-run` — гонит N аккаунтов, пишет в лог запрос + ответ LLM + score, **в BigQuery ничего не пишет**
- 🆕 **Доступ к результатам FSM-fit для PM/маркетинга** — **8h**
  - Доступ к таблице в BigQuery для PM/маркетинга (через IAM)
  - Как объединять метрики и результаты LLM в удобный вид — выбрать один из подходов:
    - View в BigQuery (JOIN `account_metrics` ⋈ `account_fsm_fit` по `account_id` при каждом запросе) — всегда свежие данные, ничего лишнего не храним, но JOIN каждый раз
    - Materialized view — читается быстрее, но с задержкой обновления и ограничениями (не все JOIN'ы поддерживаются)
    - Денормализованная таблица — писать в FSM-fit сразу с приджойненными метриками; читается быстрее всего, но дублирует данные и может расходиться, если метрики обновились, а FSM-fit ещё нет
    - Только документация + готовый запрос — каждый потребитель пишет JOIN сам по шаблону; ничего лишнего ставить не нужно, но потребители могут разойтись в запросах
  - Документация: что значит каждая колонка, как читать (короткий README рядом с таблицей/view)
- 🆕 **API для получения результата анализа по аккаунту (Tofu.AI.Backend + BFF)** — **4h**
  - В `Tofu.AI.Backend`: gRPC-метод `GetAccountAnalysis(account_id)` — возвращает FSM-fit (`score`, `tier`) + список рекомендованных офферов (`recommended_offers`, уже лежит в `account_fsm_fit.recommended_offers`); читает из BigQuery либо из локального кеша, если запрос в BigQuery на каждый вызов окажется дорогим
  - В `Invoices.Backend` (BFF): REST-эндпоинт для фронта/мобильных, пересылает запрос в gRPC + проверяет права (кто может смотреть какой аккаунт)
  - Контракт ответа: какие поля отдаём наружу (`score`, `tier`, `recommended_offers[]`, `reasoning`?), а какие прячем (внутренние версии — `prompt_version`, `model_id` и т.п. — это для отладки, клиенту не нужно)
  - Формат `recommended_offers`: список объектов с `offer_id`, `title`, `priority` / `rank`? — окончательную форму договариваем с PM в рамках сабтаска «Зафиксировать v1-сценарий»
  - Что отдаём, если результата ещё нет: `null` / 404 / заглушка со `status: pending` — определяем заранее
- ⚪ **Деплой в staging и end-to-end rollout test**

## Phase 2 — улучшения и расширение

- **Конфигурируемость** — вынос из кода в конфиг/storage того, что в Phase 1 зашито:
  - Промты — менять без релиза, версионировать отдельно от кода
  - Правила FSM-fit (`Analyses.Domain/Rules/FsmFitRule.cs`) — rule engine или DSL
  - Параметры анализа: confidence threshold, набор распознавателей Presidio, `model_id`, температура и т.п.
- **Инструмент для тестирования промтов и правил** (training/eval-цикл) — чтобы тюнить промты и правила без релиза
  - Размеченная выборка аккаунтов с «правильным» ответом (FSM-fit `tier` + ожидаемый набор офферов) — размечает PM/маркетинг
  - Прогон версии промта/правил на этой выборке → метрики качества (precision/recall по `tier`, совпадение офферов, доля адекватного `reasoning`)
  - Сравнение версий: запустить v1 и v2 на одной выборке, посмотреть, где результат отличается, и насколько — без выкатки в прод
  - Проверка `reasoning`: глазами или автоматически через LLM-as-judge
  - Опирается на «Конфигурируемость» — когда промты и правила вынесены наружу, такие итерации становятся быстрыми
- **Версионирование на уровне таблицы** (миграции схемы, не только версии записи)
- **Обработка расхождения версий метрик и анализа** — продумать поведение, когда метрики обновились до новой версии, а сохранённый анализ — ещё на старой (или наоборот). Варианты: автоматически пересчитать анализ; пометить запись как устаревшую; отдавать потребителю с пометкой «версии разошлись»; ничего не делать (положиться на следующий цикл пересчёта). Выбор зависит от того, насколько критично PM/маркетингу видеть только согласованные пары
- **Альтернативы Hangfire** (Cloud Run Jobs / Workflows) — пересмотреть, если упрёмся в потолок HF (память воркера, нет координации между подами, нужны более жёсткие SLA)
